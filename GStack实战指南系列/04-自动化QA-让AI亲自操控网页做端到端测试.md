# 04-自动化 QA：让 AI 亲自操控网页做端到端测试

> 🤖 Vibe Coding 最难的挑战就是全链路跑通。当你用 AI 写完前端页面和后端接口，你如何验证整个产品流程没有中断？在 GStack 体系里，`/qa` 和 `/browse` 给出了让许多开发者惊呼的终极答案：**让模型亲自打开浏览器点点看。**

---

## 1. 为什么传统的 AI 无法做端到端测试？

如果不使用 GStack 等技能工作流，普通的 Claude 只能帮你写测试脚本（比如写 `.test.ts`）。
它**自己是看不见页面的**。即使页面按钮错位了、弹窗不出来，它依旧认为自己的代码“完美无瑕”。

GStack 引入了 **Computer Use（计算操控）** 概念的变体：
它内置了基于 **Playwright / Puppeteer** 这类无头浏览器的控制能力。

---

## 2. 认识 `/browse` 与 `/qa` 的大招组合

### 步骤一：开启持久化浏览器 (`/browse`)

普通的 cURL 请求或者抓包，在遇到需要登录、填写表单验证的网站时统统失效。
GStack 要求使用 `/browse`：

```bash
/browse 访问 http://localhost:3000，并给我讲讲页面的初始状态。
```

当这条指令下达，Claude 会开启一个实际的 Chromium 浏览器进程实例（带 Cookie 持久化），渲染页面后捕捉结构。
你会看到它回复：“我看到了一个大标题，以及右上角的一个红色的‘登录’按钮。”

### 步骤二：让它“动”起来 (`/qa`)

光能看见是不够的，我们要它扮演测试工程师（QA）进行流程跑通。

```bash
/qa 前往 http://localhost:3000。
请尝试在用户框输入 "test_user"，密码框输入 "123456"，点击登录按钮，并告诉我控制台有没有飘红，最终是否跳转到了 `/dashboard`。
```

魔法时刻开始了：
由于 GStack 是具有自动工具链权限的：
1. 它会在本地浏览器用 playwright API 发送输入行为。
2. 它会主动监听浏览器的 `console.error`。
3. 它会获取新页面的 URL 是否变成了预期。

> **QA 报告（由 AI 生成）**：
> ❌ **测试失败**。
> **现场回溯**：我在密码框输入完成后点击了登录。但是页面没有任何响应，URL 也没有跳转。
> **抓包发现**：浏览器的控制台打印了：`TypeError: undefined is not an object (evaluating 'res.token')`。
> **诊断结论**：这是因为你的后端接口没按预期的格式返回值。

---

## 3. 把繁琐的验收抛到脑后

过去，工程师测完一个页面需要无聊地点点点几十次。
现在，只要配置好了这套无头测试流程：
1. 你写完一个组件。
2. 在同一台机子上，你的应用服务进程挂在后台。
3. 把终端扔给 Claude，让它自己不断刷页面和填数据。如果没问题，它才允许你进行最后一步的发版。

这是一套不需要写几百行 `jest` 和 `cypress` 脚本，用自然语言下达命令就能实现的活灵现现的全自动端到端测试！
