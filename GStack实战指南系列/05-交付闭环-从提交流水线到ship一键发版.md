# 05-交付闭环：从提交流水线到 `/ship` 一键发版

> 📦 当一切尘埃落定，那些为了发版（Release）需要繁重合并分支、跑 CI/CD 流程、写 Commit Message 甚至发版日志的日子，都被浓缩进了一个单词：`/ship`。

---

## 1. 原来的我们在合并代码时有多痛？

当我们辛苦用 AI 做了几天功能之后，发版的时候常常面临几个琐碎又容易出错的环节：
1. 未提交的文件散落各处，忘记 `git add`。
2. 胡乱随便填写的 `git commit -m "fix/update"` 让未来的团队无法回溯。
3. 在没有先跑一遍本地单测就往远端推（Push）导致主线全红。
4. 创建 PR 时懒得写那几十行的“此次变更影响了什么”。

在 GStack 体系里，软件交付不再是一门手工活，这就是最后一个压轴技能。

---

## 2. 解析：`/ship` 究竟在幕后做了什么？

当你在 Claude Code 的终端敲出了：

```bash
/ship 我们做完了！提交这个版本并打个 PR 吧。
```

GStack 会瞬间开启一个极其严格的发版流水线。这个流程非常具有 Google 或硅谷大厂的出海规范风范：

1. **自动 Diff 生成**：读取所有变更文件，梳理出变更地图。
2. **生成语意化 Commit**：不再是敷衍的 “Update”。它会用标准的符合 Conventional Commits 规范的词缀（如 `feat(auth): add web3 modal`、`fix(ui): resolve z-index issue`）自动整理极其漂亮的代码历史！
3. **跑通测试覆盖（可选）**：如果是完整的环境，它会主动执行你项目里规定的全量本地测试 `npm run test`。此时如果炸了，它会直接打回并强迫你修复。
4. **拉取远程规避冲突**：它会帮忙处理 `git pull` 与基础的分支重置保证版本线一致。
5. **推向远程且创建 PR（对接 GitHub API）**：它会顺水推舟将本地分支推上仓库，调用接口生成结构极其严谨、连前因后果和测试表现都自动写得一清二楚的 Pull Request (PR)。

---

## 3. 从零到一的壮举：一个人就是一支军队

至此，在这个系列里，我们把 GStack 这本《虚拟开发团队操作手册》全盘打通了。
回顾整个宏大的工作流：

💡 **/plan-ceo-review** -> 让你不忘初心，守住最小 MVP。
📝 **/plan-eng-review** -> 让你的系统有图纸，不拉跨。
🛠️ **(编写极简代码)** -> AI 根据蓝图进行开发。
🛡️ **/review** -> 对抗性审查把问题杀死在萌芽。
🤖 **/qa** -> AI 自己用浏览器充当测试工程师点通整条业务流程。
🚀 **/ship** -> 撰写完美史稿，向全世界发布你的大招。

有了 GStack 这套技能包，“让 AI 帮我写代码”这句话，正式进阶为了更为高级的范式：
**“让 AI 帮我运营并交付这款数字产品。”**
