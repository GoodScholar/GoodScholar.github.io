# 02-产品与架构：用 CEO 与技术总监视角锁定项目边界

> 💡 俗话说，写代码只占开发的 20%，剩下 80% 的时间在弄清楚“到底该写什么”以及“写错了怎么重构”。GStack 提供的第一道防线，就是把这两个痛点在写代码前锁死。

---

## 1. 为什么要让 AI 扮演 CEO？

在全栈独自开发的场景下，极易陷入**严重的范围蔓延（Scope Creep）**。
比如，你最初只想做个任务大屏，结果写着写着，就开始加入黑夜模式切换、加入高级动画库、甚至开始折腾一个极其复杂的自定义日历组件。
一周过去了，你的核心业务都还没跑通。

Garry Tan 加入 `/plan-ceo-review`（还有它的姊妹指令 `/office-hours`）的作用，就在于给你一棒子。

### 实战演示：`/plan-ceo-review`
当你想增加一个功能的时候，通过这个快捷斜杠启动：

```bash
/plan-ceo-review 
我想在文章页面加上一个通过 Web3 钱包打赏的功能，用户可以绑定小狐狸（MetaMask）。
```

此时 Claude 会被 GStack 转换为“严肃的 CEO”。它可能会回怼你：
> **CEO 评审意见**：
> ❌ **拒绝或搁置**。
> **理由**：你的当前站点日活用户不足 100 人。加入 Web3 钱包组件会引入臃肿的 `ethers.js` 库，大幅提高编译时间和潜在 Bug 风险。在这个阶段，开发一个简单的“赞赏码图片”才是符合核心 MVP 验证的方法。

通过这种强制性的对抗推演，它会帮你把精力拉回主干线。

---

## 2. 让大模型去写代码前：先做技术审查

无数“屎山”诞生的根源是向 AI 下达诸如：“给这页加上权限管理”的结果。
一旦涉及到状态提升、数据穿透，它就开始给你瞎引文件了。
此时你需要 `/plan-eng-review`。

它会让 Claude 变为经验丰富的 **Tech Lead（技术总监）**。

### 实战演示：`/plan-eng-review`
紧接刚才的沟通结果，假设 CEO 批准了加一个简单的赞赏弹窗。

```bash
/plan-eng-review 
按照确定的 MVP 需求，我们需要一个基于状态控制的全局赞赏 Modal。请规划技术方案。
```

它不会去写这几百行代码。相反，它会给出：
1. **组件流转图**：明确哪个顶层 Provider 控制 `isModalOpen` 的 State。
2. **Interface 规约**：强制你（或未来的它）用哪些确定的 Props 和接口。
3. **防坑警示**：它会提醒“你要注意这个 Modal 的 `z-index` 防止被现有的顶栏遮挡”。

---

## 3. 把约束作为护城河

经过了 CEO 视角明确**“要做什么”**，经历了 Tech Lead 视角规定**“要怎么做（架构边界）”**。

这时的你的项目状态就好像已经经过了评审会的洗礼。
当你在接下来的指令中，叫 Claude 开始老老实实写 React 和 CSS 代码时，只要引用刚才建立好的那份 review 文档。**不管你是用 Cursor 还是 Claude Code，AI 生成烂代码的几率为直线跳崖式下降。**

这便是 GStack 对工程纪律改造的灵魂。
