# 03-对抗性审查：深入探索 `/review` 代码深度审计防线

> 🛡️ 很多初学者误以为大模型生成的代码是完美的。资深开发者知道，AI 生成的代码往往充满了“看起来很对”的隐患。在你的代码合入主干前，`/review` 是一道你必须要过的鬼门关。

---

## 1. 什么是“对抗性审查”？

普通的提示词：
> “帮我看看这段代码有没有 Bug？”
AI 通常会回答：“这段代码看起来很好，只是一些语法可以优化……”这是一种讨好型人格的回复。

GStack 的 `/review` 指令并不是为了“看字面语法”。它的系统设定是：**这批代码是对安全性、扩展性和业务鲁棒性有直接威胁的危险品。**
GStack 会扮演一个极其尖酸刻薄的高级资深代码审计员。它会主动向你发起攻击：
▪️ “你的输入没有做防 XSS 注入处理。”
▪️ “在这个 `useEffect` 里，你没有考虑竞态条件（Race Condition）。”
▪️ “这行会导致 N+1 查询，当数据量超过 1000 时必定超时。”

---

## 2. /review 的工作原理

当你在 Claude Code 中输入 `/review` 时，GStack 背后的工作流是如下的：

1. **获取 Diff**：它会立刻读取你当前 Git 工作树下未提交（Uncommitted）或未合并分支的 Diff 变更。它不看无关文件，只盯死你刚刚写的部分。
2. **结合架构意图**：它会回溯你在 `/plan-eng-review` 阶段所写的工程规约和约定，检查你这批代码是否偏离了原始架构方向。
3. **出具报告**：生成一份包含【致命问题】、【性能隐患】、【代码坏味道】的结构化 Markdown 报告。

### 实战示例

```bash
/review 请检查刚才添加的赞赏弹窗逻辑。
```
**AI 的无情打击：**
> ⚠️ **内存泄漏警告**：你在 `App.tsx` 的 `useEffect` 中监听了窗口滚动的事件来控制 Modal 的显示，但你忘记在组件卸载（unmount）时 `removeEventListener`。
> 🔄 **优化建议**：我已提供修复上述内存泄漏的 Diff。是否允许我直接将补丁应用到你的工作区？

（此时如果是 Claude Code 环境，输入 `yes` 即可由 AI 自动完成代码替换。）

---

## 3. 把 Review 编织进你的提交习惯

不要等到你的功能完全做完了、写了上千行再去跑 `/review`。那时候的隐患往往盘根错节，让 AI 都无法准确追溯根因。

**GStack 最佳纪律要求**：
每一个功能的小型闭环（例如你只花了 1 小时写好了数据库查询部分和其数据层接口）只要一完成，立刻 `git add` 并执行 `/review`。

这就像是一位不会累的 Senior 开发，在你看不到的地方，时刻守护着你的底线。
