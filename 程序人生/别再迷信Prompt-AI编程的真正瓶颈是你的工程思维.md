# 别再迷信 Prompt — AI 编程的真正瓶颈是你的工程思维

> 🧠 你以为 AI 编程的核心技能是"写好 Prompt"？不。Prompt 只是触发器，工程思维才是操作系统。

---

## 一个让我警醒的夜晚

上个月，我花了一整晚用 Claude Code 开发一个功能——很简单，一个带搜索和筛选的列表页。

Prompt 写得很漂亮。我甚至参考了社区流行的"结构化提示词模板"，角色设定、约束条件、输出格式一应俱全。AI 生成的代码也确实"能跑"。

但问题出在第二天。

当我需要加一个排序功能时，发现 AI 前一晚写的状态管理逻辑根本没法扩展——搜索关键词、筛选条件、分页状态全部混在一个巨型 `useState` 里，耦合到像一碗开胶的意大利面。我不得不推倒重来。

那一刻我意识到：**AI 忠实地执行了我的 Prompt——问题是，我的 Prompt 本身就是面条代码级别的思考质量。**

---

## "Prompt 工程师"是一个伪命题

2024 年到 2025 年，整个行业都在鼓吹"Prompt 工程"。"学会写 Prompt 就能 10x 提效"，"Prompt 模板大全，一键生成完美代码"——这些说法在某种程度上没有错，但它们掩盖了一个更重要的事实：

> **Prompt 的上限，取决于你对问题本身的理解深度。**

你不可能用一句精彩的 Prompt 让 AI 生成一个你自己都没想清楚的系统架构。就像给一个顶级厨师再好的食材，如果你的菜谱是"做个好吃的东西"，出来的结果也是随机的。

### 两种人用 AI 的区别

**第一种人（关注 Prompt 技巧）：**

```
"你是一个资深 Flutter 工程师。请用 Riverpod 实现一个用户管理模块，
包含列表、搜索、分页功能。代码要简洁、可维护、有注释。"
```

生成的代码看起来不错，但：
- 状态粒度没人定义，AI 按自己的理解随意拆分
- 错误处理策略是 AI 即兴发挥的
- 缓存策略和数据一致性完全没考虑
- 当你想加需求时，改一个地方要连带改五个

**第二种人（关注工程思维）：**

在写任何 Prompt 之前，他先在脑子里过了一遍：

```
这个用户管理模块的核心问题是什么？
├── 状态管理：搜索/筛选/分页是三个独立状态，可以组合但不能耦合
├── 数据流向：API → Repository → StateNotifier → UI，单向
├── 错误边界：网络错误走 retry，业务错误走 toast，数据错误走 fallback
├── 扩展点：未来会加排序和批量操作，状态层要预留空间
└── 测试策略：Repository 层 mock，StateNotifier 层 unit test
```

然后他的 Prompt 不再是"帮我实现一个用户管理模块"，而是分成 3-4 个精确的步骤，每一步都有明确的输入和约束。

**同样是用 AI，第一种人在碰运气，第二种人在做工程。**

## AI 编程的三个真正瓶颈

经过大半年的密集使用，我总结出 AI 编程效果好坏的三个核心变量。没有一个是"Prompt 写得好不好"。

### 瓶颈一：你拆解问题的能力

AI 最擅长的事情是：**在一个小而清晰的上下文里，完成一个边界明确的任务。**

AI 最差的事情是：**在一个大而模糊的上下文里，做一个边界不清的事情。**

这意味着，你喂给 AI 的任务颗粒度，直接决定了产出质量。

```
❌ 差的任务拆解：
"帮我做一个聊天功能"
→ AI 会生成一坨不可维护的代码

✅ 好的任务拆解：
Step 1: "设计聊天消息的数据模型，需要支持文字、图片、语音三种类型"
Step 2: "实现消息列表的 UI 组件，支持虚拟滚动和上拉加载"
Step 3: "实现 WebSocket 连接管理，包含重连和心跳机制"
Step 4: "将三者集成，处理发送和接收的状态同步"
→ 每一步 AI 都能给出高质量的代码
```

这个能力和 Prompt 语法没有半毛钱关系。它是纯粹的**工程分解能力**——你能不能把一个复杂系统拆成相互独立、边界清晰的子问题。

这个能力，只有真正做过工程的人才有。

### 瓶颈二：你控制上下文的能力

大模型有上下文窗口限制，但真正的限制不是 token 数量——是**噪声比**。

我见过很多人把整个项目的代码一股脑塞给 AI，然后抱怨 AI"不理解我的项目"。但如果你给一个人类同事 50 个文件让他同时看，他也会懵。

真正的高手在做一件事：**精确地控制 AI 在任何一个时刻看到什么、不看到什么。**

```
原则：
1. 当前任务只需要用到 3 个文件，就只给 AI 看这 3 个文件
2. 与当前任务无关的代码逻辑，宁可用文字描述接口契约也不要放源码
3. 跨模块的上下文靠「接口定义 + 行为描述」传递，不靠源码
4. 每完成一个子任务，清理上下文再进入下一个
```

这就像给新同事做 onboarding——你不会一次把所有文档丢给他，而是根据他当前要做的事情，给他恰到好处的信息量。

**AI 的"智力"并没有在不同 Session 之间变化。变化的只有你给它的上下文质量。**

### 瓶颈三：你验证结果的能力

这是我认为最被低估的一点。

很多人把 AI 当搜索引擎——搜完就用，不验证。但 AI 生成的代码和搜索引擎出来的答案有一个本质区别：**它看起来总是对的，即使它是错的。**

AI 不会在代码里写"我不确定这里的实现是否正确"。它会用非常自信的语气、漂亮的代码格式、完整的注释，给你一段可能有微妙逻辑错误的代码。

所以，验证能力成了分水岭：

| 验证层级 | 做法 | 成本 |
|---------|------|------|
| Level 0：不验证 | "AI 写的应该没问题吧" | → 线上出 Bug 的成本 |
| Level 1：人肉 Review | 逐行读 AI 生成的代码 | → 时间成本高 |
| Level 2：自动化测试 | 让 AI 同步生成单测 | → 中等 |
| Level 3：闭环验证 | 测试 + 类型检查 + 运行验证 + 边界 Case | → 低（一次投入，持续受益）|

Level 3 的人会在 AI 生成代码的同时，要求它：
1. 生成单元测试
2. 生成边界 Case 列表
3. 标注"我不确定的地方"
4. 提供替代方案的 trade-off 分析

**这不是 Prompt 技巧。这是工程纪律。**

## 工程思维是可以训练的

好消息是，工程思维不是天赋，它是可以刻意训练的。以下是我正在用的"训练菜单"：

### 练习一：在写 Prompt 之前先写分解

每次需要 AI 帮忙之前，先在纸上（或脑子里）画一个树状结构：

```
我要做的事
├── 子任务 A（输入是什么？输出是什么？依赖什么？）
├── 子任务 B（和 A 的边界在哪？接口是什么？）
└── 子任务 C（这个任务自身的复杂度在哪？）
```

如果你画不出这棵树——说明你对问题的理解还不够，这时候给 AI 的任何 Prompt 都是在碰运气。

### 练习二：复盘 AI 犯的每一个错

AI 生成了一段有 Bug 的代码？别急着让它重新生成。先问自己：

- 是我的需求描述有歧义吗？
- 是我给的上下文不够吗？
- 是我没有指定某个关键约束吗？
- 如果我是 AI，看到这段输入，我也会这样理解吗？

90% 的情况下，你会发现问题出在你这边。每一次复盘都是在训练你"为 AI 思考"的能力。

### 练习三：假装你在带一个聪明但没经验的实习生

这是我见过最好的心智模型。

AI 就像一个编程能力极强、但对你的项目完全陌生的实习生。他能写出教科书级别的代码，但缺的是——对你项目特有的上下文理解、对你团队特有的规范了解、对你业务特有的坑的经验。

你会怎么带这个实习生？

- 你不会把整个项目扔给他说"你看着写"
- 你会先给他讲清楚架构和规范
- 你会把任务拆细，每次给一个，做完验收再给下一个
- 你会在关键决策点亲自把关

**把这套带人的方法论用在 AI 上，效果立竿见影。**

## 一个反直觉的结论

AI 编程工具越来越强，但**有能力用好它的人反而越来越少**。

因为核心瓶颈不在工具端，在人端。Prompt 写法可以零成本复制粘贴，但工程思维——问题分解、上下文控制、验证闭环——这些需要真正的项目积累和刻意练习。

所以，与其花时间去收藏"2026 年最全 Prompt 模板"，不如：

1. 认真做一个完整项目，体会"AI 写了但不好用"的痛点
2. 刻意练习任务分解，直到你能在 30 秒内把一个需求拆成 5-8 个子任务
3. 建立自己的验证习惯，永远不无脑信任 AI 的产出

Prompt 会过时，模型会升级，工具会换代。

**工程思维不会。它是你在 AI 时代唯一不折旧的资产。**
