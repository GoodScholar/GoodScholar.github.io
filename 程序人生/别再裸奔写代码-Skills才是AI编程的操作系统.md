# 别再裸奔写代码 — Skills 才是 AI 编程的「操作系统」

> 🧠 没有 Skills 的 AI 编程，就像一台没装操作系统的裸机——硬件再强，你也只能一条一条地输入机器码。

---

## 一个真实的对比实验

上周，我做了一个实验。

同一个需求——给 Taro 项目新增一个带表单验证的设置页面——我用两种方式让 AI 来完成：

**方式 A：裸奔模式**

打开编辑器，对着 AI 说："帮我写一个设置页面，包含昵称、头像、手机号三个表单项，要有校验逻辑。"

AI 写了一版。能跑，但问题一堆：
- 组件用了 `@antmjs/vantui`——我项目上个月已经迁移到 `@taroify/core` 了
- CSS 类名风格和项目里其他页面完全对不上
- 表单验证逻辑是 AI 即兴写的，和我封装好的 `useFormValidation` 完全重复
- 页面布局没用我统一的 `Layout` 组件

修修补补花了 40 分钟，比自己手写还慢。

**方式 B：Skills 加持模式**

我的 AI 编辑器已经配置好了一套 Skills：项目架构规范、组件使用约定、代码风格规则、以及一个 `page-scaffold` Skill（页面脚手架模板）。

同样的需求，AI 直接生成了：
- 基于 `@taroify/core` 的表单组件
- 自动使用 `Layout` 组件包裹
- 复用已有的 `useFormValidation` Hook
- CSS 命名遵循项目的 BEM 规范

**不需要改一行代码。** 整个过程 3 分钟。

这就是"裸奔"和"有系统"的区别。

## 什么是 Skills？先说清楚

如果你还不了解 Skills，用一句话解释：

> **Skills 是一组预定义的规则和指令，让 AI 在编程时自动遵循你的项目规范、技术选型和代码风格。**

你可以把它理解成——给 AI 装的"操作系统"。

| 概念 | 类比 | 作用 |
|------|------|------|
| 大模型 | CPU/硬件 | 提供底层智能，能理解和生成代码 |
| Prompt | 你敲的一条命令 | 告诉 AI 这次要做什么 |
| Skills | 操作系统 | 定义 AI 的默认行为、规范、上下文 |
| Workflows | 应用程序 | 基于 Skills 编排的自动化流程 |

没有操作系统的电脑，每次开机你都得从零开始告诉它怎么读磁盘、怎么显示画面、怎么响应键盘。有了操作系统，这些底层的事情都被封装好了，你只需要关注业务。

**Skills 对 AI 编程做的，是完全相同的事。**

## 裸奔写代码的五个代价

很多人觉得 Skills 是"锦上添花"，不弄也行。但当你认真盘点裸奔的代价，你会发现这不是"锦不锦"的问题——这是成本问题。

### 代价一：重复说明，每次都像新入职

没有 Skills，你每次和 AI 对话都要重新解释：
- "我们项目用 Taro + React + TypeScript"
- "组件库用 @taroify/core，不要用 antd"
- "CSS 用 BEM 命名"
- "状态管理用 Zustand"

这就像你有一个实习生，但他每天早上都失忆了。你不得不每天重新做一次 onboarding。

Skills 解决的就是这个——**一次定义，永久生效**。

### 代价二：风格漂移，代码库越来越像拼图

AI 每次生成的代码风格是随机的。今天驼峰，明天下划线；这个文件用 `async/await`，那个文件用 `.then()`；同一个功能在 A 页面是一种写法，在 B 页面是另一种。

一个月下来，你的代码库看起来像十个不同的人写的——因为确实如此，每次 AI 的"人格"都不一样。

### 代价三：轮子重复造，已有封装被无视

这是让我最抓狂的。

我明明封装了 `useRequest`、`useFormValidation`、`Layout`、`CellItem` 这些通用组件和 Hook，但裸奔模式下 AI 完全不知道它们的存在。它会非常自信地从零开始造一个功能一模一样但实现完全不同的版本。

结果就是：你的项目里出现了两套表单验证逻辑、三种网络请求封装、四个不同的列表布局实现。

### 代价四：架构腐化，规范只存在于你脑子里

每个有经验的开发者脑子里都有一套"在这个项目里，事情应该怎么做"的规则：

- 页面组件放在 `pages/` 下，公共组件放在 `components/` 下
- API 请求统一走 `services/` 层，不要在组件里直接调
- 新页面要注册路由到 `app.config.ts`
- 错误处理统一用 `Toast` 而不是 `alert`

这些规则没有人会每次写 Prompt 时列一遍。但 AI 不知道，所以它会以各种"合理但不符合你项目规范"的方式来组织代码。

一开始你可能不在意，但架构腐化是累积性的——等你发现的时候，重构成本已经指数级上升了。

### 代价五：时间浪费在"擦屁股"而不是"创造价值"

最终，所有上述问题汇聚到一个结果：**你的时间大量花在修改 AI 生成的代码上，而不是思考真正的业务逻辑。**

这完全违背了 AI 编程的初衷——我们是想用 AI 来提效，不是来创造更多需要修改的代码。

## Skills 的四层架构

一套好的 Skills 体系应该分四层，从上到下依次是：

```
┌─────────────────────────────────────────┐
│       Layer 4: Workflow Skills          │
│  (编排层：多步骤自动化工作流)              │
├─────────────────────────────────────────┤
│       Layer 3: Practice Skills          │
│  (实践层：TDD、Code Review、调试流程)      │
├─────────────────────────────────────────┤
│       Layer 2: Convention Skills        │
│  (规范层：代码风格、命名、目录结构)          │
├─────────────────────────────────────────┤
│       Layer 1: Context Skills           │
│  (上下文层：技术栈、依赖、项目基本信息)      │
└─────────────────────────────────────────┘
```

### Layer 1：上下文层 — 让 AI 知道你是谁

这是最基础的一层，告诉 AI 你的项目"长什么样"：

```markdown
# 项目上下文 Skill

## 技术栈
- 框架：Taro 4.x + React 18
- UI 组件库：@taroify/core（不要使用 @antmjs/vantui）
- 状态管理：Zustand
- 样式方案：CSS Modules + BEM 命名
- 语言：TypeScript（strict 模式）

## 目录结构
- src/pages/       → 主包页面
- src/pages2/      → 分包2页面
- src/components/  → 公共组件
- src/services/    → API 请求层
- src/hooks/       → 自定义 Hook
- src/stores/      → Zustand Store

## 已有公共组件
- Layout：页面统一布局容器，所有页面必须使用
- CellItem：统一的列表项组件
- DynamicCard：动态内容卡片
```

别小看这一层。仅仅是这些基本信息，就能让 AI 生成代码的"可用率"从 30% 提升到 70%。

### Layer 2：规范层 — 让 AI 写出来的代码"像你写的"

```markdown
# 代码规范 Skill

## 命名规范
- 组件文件：PascalCase（如 UserProfile.tsx）
- Hook 文件：camelCase，以 use 开头（如 useAuth.ts）
- 样式文件：与组件同名，使用 .module.scss 后缀
- 常量：UPPER_SNAKE_CASE

## 组件规范
- 函数组件使用箭头函数导出：export const Component = () => {}
- Props 类型单独定义，命名为 ComponentNameProps
- 必须使用 React.memo 包裹纯展示组件

## Git 规范
- feat: 新功能
- fix: Bug 修复
- refactor: 重构
- style: 样式调整
- chore: 工具链/配置修改
```

这一层的核心是**一致性**。AI 不再"即兴发挥"，而是严格按照你的规范来。

### Layer 3：实践层 — 让 AI 遵循工程纪律

```markdown
# 开发实践 Skill

## Code Review 规则
在提交任何代码修改前，必须检查：
1. 是否引入了新的依赖？如果是，需要说明理由
2. 是否影响了现有组件的 API？如果是，需要列出影响范围
3. 是否有遗漏的错误处理？
4. 是否有未清理的 console.log？

## 调试流程
遇到 Bug 时，按以下顺序排查：
1. 先确认问题的可复现路径
2. 检查是数据问题还是 UI 问题
3. 缩小范围到最小可复现单元
4. 修复后验证不影响其他功能
```

这一层把"老手怎么做事"的经验固化下来，让 AI 也按照高标准执行。

### Layer 4：编排层 — 让 AI 自动完成多步骤任务

```markdown
# 新页面创建 Workflow

当用户要求创建新页面时，按以下步骤执行：

## Step 1: 创建页面文件
在 src/pages/ 或对应分包下创建页面目录和 index.tsx

## Step 2: 创建样式文件
在同目录下创建 index.module.scss

## Step 3: 注册路由
在 app.config.ts 中添加页面路径

## Step 4: 创建 Service
如果页面有 API 请求，在 src/services/ 下创建对应的 service 文件

## Step 5: 验证
确认页面可以正常导航和渲染
```

这一层实现了真正的**自动化**——一个指令，多步联动。

## 从裸奔到有系统：我的 Skills 演进之路

回顾我这半年的 AI 编程之路，Skills 体系是逐步长出来的，大致经历了三个阶段：

### 阶段一：README 时代（效率提升 30%）

最初我只是在项目根目录放了一个 `CODING_RULES.md`，里面写了基本的技术栈和代码风格。

每次和 AI 对话时手动 @ 这个文件，告诉它"按照这个规范来"。

优点：比裸奔好了很多，AI 起码知道用正确的组件库了。
缺点：要手动引用，经常忘记；文件越写越长，AI 处理大文件效率下降。

### 阶段二：分散 Skills 时代（效率提升 60%）

后来我把那个巨大的 README 拆成了多个小 Skill 文件——技术栈一个、代码规范一个、组件清单一个。

根据不同任务加载不同的 Skills。做组件时加载组件规范，做页面时加载页面模板。

优点：上下文更精准，AI 的输出质量明显提升。
缺点：需要我判断加载哪些 Skills，有认知负担。

### 阶段三：系统化 Skills 时代（效率提升 300%）

现在的状态。我用了 Superpowers 和 GStack 这样的 Skills 框架来管理所有的技能定义。

框架自动判断当前任务需要哪些 Skills，自动注入上下文。我只需要像之前一样描述需求，背后的 Skills 系统自动运转。

这就是"操作系统"的比喻成立的地方——**你不需要每次都告诉操作系统怎么管理内存，它自己知道该怎么做。好的 Skills 系统也是如此。**

## 实战：5 个你今天就能用的 Skills

别被"系统化"吓到。你不需要一上来就搭建完整的 Skills 体系。以下五个 Skill 是投入产出比最高的起步选择：

### 1. 技术栈声明 Skill

```markdown
# tech-stack.md
本项目使用以下技术栈，所有生成的代码必须遵循：
- [你的框架和版本]
- [你的组件库]
- [你的状态管理方案]
- [你的样式方案]
禁止使用：[列出常见的错误选择]
```

**效果**：消灭 80% 的"用错库"问题。

### 2. 组件清单 Skill

```markdown
# components.md
项目中已有以下公共组件，优先使用而非重新实现：
- Layout：页面容器，props: { title, showBack, showNav }
- CellItem：列表项，props: { label, value, onClick, arrow }
- ...
```

**效果**：消除重复造轮子。

### 3. 代码风格 Skill

```markdown
# code-style.md
- 使用箭头函数导出组件
- Props 类型以 Props 后缀命名
- 变量命名使用 camelCase
- 文件名使用 PascalCase
```

**效果**：保持代码库一致性。

### 4. 错误处理 Skill

```markdown
# error-handling.md
- 网络错误：使用 Toast.fail() 提示用户
- 表单验证：使用内联错误提示
- 空状态：使用 Empty 组件展示
- 加载状态：使用 Skeleton 骨架屏
```

**效果**：统一异常处理策略。

### 5. Git 提交 Skill

```markdown
# git-commit.md
提交信息格式：<type>(<scope>): <description>
- type: feat|fix|refactor|style|chore|docs
- scope: 模块名称
- description: 中文描述，简洁明了
示例：feat(user): 新增用户设置页面
```

**效果**：Git 历史清晰可读。

## Skills vs Prompt：不是替代，是升维

有人问："有了 Skills，是不是就不用写 Prompt 了？"

不是。Skills 和 Prompt 是两个维度的东西：

```
Skills（横向） × Prompt（纵向） = AI 编程的产出质量

Skills 定义「怎么做」—— 规范、约束、上下文
Prompt 定义「做什么」—— 具体任务、需求、目标
```

**没有 Skills 的 Prompt**，就像在白纸上画建筑——每次都要从地基画起。

**有 Skills 的 Prompt**，就像在蓝图上标注——地基已有，你只需要指明要加哪面墙。

这不是效率的线性提升，是指数级的。因为 Skills 帮你消除了每次 Prompt 里 70% 的重复说明，让 AI 的理解力集中在真正重要的 30% 上。

## 一个反直觉的预测

2026 年到 2027 年，AI 编程领域会出现一个明显的分化：

- **一类开发者**：每天花大量时间"调教" AI——反复修改 Prompt、手动纠正代码风格、处理组件库用错了的问题。他们的效率可能只比不用 AI 快 20%。

- **另一类开发者**：前期花了几天搭建 Skills 体系，之后每次写代码只需要描述业务需求。他们的效率是不用 AI 时的 3-5 倍。

差距不在智商，不在经验年限，甚至不在 AI 工具选择——**差距在于你有没有一个"操作系统"。**

所以，我的建议很简单：

1. **今天就开始**，哪怕只写一个 `tech-stack.md`
2. **每次踩坑就沉淀**，AI 用错了什么，就补一条 Skill 规则
3. **定期重构**，Skills 和代码一样需要迭代优化
4. **分享给团队**，好的 Skills 是可以复用的团队资产

别再裸奔了。给你的 AI 装上操作系统，你会发现——**同一个 AI，换一套 Skills，就像换了一个人。**
