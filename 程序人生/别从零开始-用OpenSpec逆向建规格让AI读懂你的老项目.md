# 别从零开始 — 用 OpenSpec 逆向建规格，让 AI 读懂你的老项目

> 🔍 你有一个跑了两年的项目，2 万行代码，零文档。你想让 AI 帮你加个功能，它说："这个项目是什么架构？用了什么组件库？路由怎么组织的？" —— 欢迎来到存量项目的 AI 困局。

---

## 一个让人抓狂的场景

周一早上，产品经理甩过来一个需求："在圈子详情页加一个成员列表，要分页加载。"

你打开 Claude Code / Gemini CLI，输入：

```
帮我在圈子详情页加一个成员列表组件，支持分页加载。
```

AI 开始工作了。5 分钟后你拿到代码，一看——

- 路由跳转用了 `navigator.push()`，你项目里封装的是 `Taro.navigateTo`
- 网络请求直接用了 `fetch`，你明明封装了 `useRequest` Hook
- 列表用了原生 `ScrollView`，你项目里所有列表统一用 `VirtualList` 组件
- 样式写了内联 `style`，你整个项目用的是 CSS Modules

AI 写的每一行代码，**技术上都是正确的，但在你的项目语境里全是错的。**

修了 40 分钟，还不如自己写的快。

这不是 AI 的问题。是**你的项目对 AI 来说是一个黑盒**。

---

## 🎯 1. 问题的根源：存量项目 = AI 的黑盒

新项目不存在这个问题——你可以从第一行代码开始就让 AI 全程参与，它知道每一个设计决策的来龙去脉。

但 80% 的开发者面对的是**存量项目**：

| 存量项目的特征 | AI 遇到的困难 |
|:---:|:---|
| 已有大量代码（数万行） | AI 无法一次读完，只能看局部 |
| 零文档或文档过时 | AI 无法了解设计意图和架构约定 |
| 隐性约定居多 | "我们项目里不用这个库"这类知识只存在于人脑中 |
| 多次重构的历史包袱 | 同一个功能可能有新旧两套实现 |
| 团队特有的封装 | 公共组件、Hook、工具函数 AI 不知道存在 |

> **核心矛盾**：你脑子里对项目了如指掌，但 AI 每次对话都像第一天入职的实习生。

解决这个矛盾的方法不是"把整个项目喂给 AI"——上下文窗口装不下，就算装下了理解率也不高。正确的做法是：**把你脑子里的隐性知识，变成 AI 能快速读取的显性规格。**

这就是 OpenSpec 逆向建规格的核心思路。

---

## 🧩 2. 什么是"逆向建规格"

正常的 OpenSpec 工作流是：

```
先写规格 → 再写代码 → 归档规格
```

但存量项目已经有代码了，没法回到"先写规格"的起点。所以要反过来：

```
已有代码 → 让 AI 阅读代码 → 生成规格草稿 → 人工审核修正 → 存入 specs/
```

**逆向建规格 = 从代码中提炼出结构化的项目规格文件。**

这个过程不是自动化的"万能按钮"，而是一个**人机协作的知识提炼过程**：

| 步骤 | 由谁完成 | 做什么 |
|:---:|:---:|:---|
| ① 读代码 | AI | 扫描一个模块的代码，理解结构 |
| ② 生成草稿 | AI | 输出规格文件的初稿（.md 格式） |
| ③ 审核修正 | **你** | 修正 AI 理解错误的部分，补充隐性知识 |
| ④ 存入 specs/ | 你 / AI | 放到 `openspec/specs/` 目录 |

> 你审核的时间大约是 AI 读代码时间的 1/3。AI 花 10 分钟读完一个模块，你花 3 分钟看一遍修正。比你从头写文档快 5 倍。

## 🛠️ 3. 完整实战：给一个 Taro 项目逆向建规格

下面用一个真实场景演示全过程。假设你有一个 Taro + Vue3 的社交应用，已经开发了 3 个月，有用户系统、圈子系统、消息系统三个核心模块。

### Step 1：初始化 OpenSpec 目录

```bash
cd your-project
openspec init
# 选择你的 AI 工具（Claude Code / Gemini CLI / Cursor）
```

这会创建基本目录结构：

```
openspec/
├── specs/       # 规格文件（目前为空）
└── changes/     # 变更提案
```

### Step 2：选一个模块开始（别贪多）

**必须一个模块一个模块来。** 一次性让 AI 读整个项目，输出的规格质量很差——因为 AI 的注意力被稀释了。

推荐顺序：**从最核心、被依赖最多的模块开始。**

```
推荐顺序：
1. 用户/认证模块 ← 通常被所有其他模块依赖
2. 公共组件库 ← 影响所有页面的生成质量
3. 业务核心模块 ← 按重要性逐个推进
```

### Step 3：让 AI 阅读代码并生成规格草稿

对 AI 说：

```
请阅读 src/services/auth.ts、src/stores/user.ts、src/pages/login/ 
目录下的所有文件，然后为认证模块生成一份 OpenSpec 规格文件。

规格文件需要包含：
1. 模块职责描述
2. 技术方案（认证方式、Token 策略等）
3. 数据模型（关键字段和类型）
4. API 接口清单
5. 对外暴露的 Hook / Store / 工具函数
6. 和其他模块的依赖关系
```

AI 在阅读代码后，会生成类似这样的草稿：

```markdown
# 认证模块规格

## 模块职责
负责用户登录注册、Token 管理、登录态维持。

## 技术方案
- 认证方式：微信小程序静默登录 + 手机号绑定
- Token 策略：自定义 JWT Token
  - Access Token 有效期 2 小时
  - Refresh Token 有效期 30 天
- Token 存储：Taro.setStorageSync('token', ...)

## 数据模型
User {
  id: string (UUID)
  nickname: string
  avatar: string (URL)
  phone: string (可选，绑定后才有)
  role: 'user' | 'admin'
  createdAt: string (ISO 8601)
}

## API 接口
- POST /api/auth/wx-login     → 微信静默登录
- POST /api/auth/bind-phone   → 绑定手机号
- POST /api/auth/refresh       → 刷新 Token
- GET  /api/auth/profile       → 获取当前用户信息

## 对外暴露
- useAuth() Hook → { isLoggedIn, user, login, logout, refreshToken }
- useUserStore → Zustand Store，存储用户状态
- authInterceptor → 请求拦截器，自动附加 Token

## 依赖关系
- 被 [圈子模块] 依赖：创建圈子需要认证
- 被 [消息模块] 依赖：WebSocket 连接需要 Token
- 依赖 [请求模块]：使用 useRequest Hook 发起请求
```

### Step 4：审核并修正

AI 生成的草稿不会 100% 准确。你需要审核以下几点：

| 检查项 | 常见问题 | 举例 |
|:---:|:---|:---|
| 技术方案准确性 | AI 可能猜错了 Token 策略 | 你实际用的是 Session，不是 JWT |
| 隐性约定 | AI 不知道"这个字段虽然存在但不要用" | `role` 字段已废弃，改用 `permissions` |
| 依赖关系 | AI 只能从 import 推断，可能遗漏 | 消息模块通过事件总线依赖认证模块 |
| 未来规划 | AI 不知道你计划做什么 | 需要补充"计划支持三方登录" |

修正后的规格文件，存入 `openspec/specs/auth.md`。

### Step 5：逐个模块重复

按照优先级，对每个模块重复 Step 3-4：

```bash
# 最终的 specs 目录
openspec/specs/
├── auth.md           # 认证模块（第 1 个完成）
├── components.md     # 公共组件清单（第 2 个）
├── circle.md         # 圈子模块（第 3 个）
├── message.md        # 消息模块（第 4 个）
├── conventions.md    # 项目约定（代码风格、目录结构等）
└── tech-stack.md     # 技术栈声明
```

> **实战建议**：一次性完成所有模块不现实。建议每天花 20 分钟做 1 个模块，一周左右就能把核心模块的规格建完。

## 🔗 4. 逆向建完之后：Superpowers 接管

规格建完了，然后呢？

这就是 **OpenSpec + Superpowers 组合** 发挥威力的时候。

**没有规格时**的开发流程：

```
你: "加个成员列表"
AI: （读 2000 行代码，理解 60%）
    → 用错组件库
    → 重复造轮子
    → 风格不一致
你: （修 40 分钟）
```

**有规格后**的开发流程：

```
你: /opsx:propose add-member-list
    OpenSpec 读取 specs/，生成提案（自动引用 circle.md 和 components.md）

你: 按提案实施
    Superpowers brainstorming 基于已有规格讨论技术细节
    → 知道该用 VirtualList 而不是原生 ScrollView
    → 知道该用 useRequest 而不是 fetch
    → 知道该用 CSS Modules 而不是内联样式

你: /opsx:archive add-member-list
    OpenSpec 将变更归档，circle.md 更新，增加成员列表说明
```

**对比效果**：

| 维度 | 无规格 | 有规格（逆向建立后） |
|:---:|:---|:---|
| AI 理解项目耗时 | 每次 10-20 分钟读代码 | **每次 1 分钟读 specs/** |
| AI 理解准确率 | 约 60%（漏掉隐性约定） | **约 95%（规格写明了）** |
| 组件选择正确率 | 经常用错 | **几乎不出错** |
| 人工修正时间 | 30-60 分钟/次 | **0-5 分钟/次** |
| 项目知识累积 | 不累积，每次重来 | **持续累积到 specs/** |

> **关键差异**：逆向建规格是一次性投入，但收益是长期复利。每新增一个模块的规格，后续所有涉及该模块的 AI 开发都会更准确。

## 📋 5. 六种存量项目的逆向策略

不同类型的存量项目，逆向策略不同：

### 策略 A：有文档但过时的项目

```
旧文档 + 当前代码 → AI 对比差异 → 更新为 OpenSpec 格式
```

最省力。让 AI 同时读旧文档和当前代码，生成的规格草稿质量最高。

### 策略 B：完全无文档的项目

```
代码 + 你的口述 → AI 读代码 + 你补充隐性知识 → 规格文件
```

最常见。按本文 Step 3-4 的流程走。

### 策略 C：有单元测试的项目

```
测试文件 → AI 从测试推断接口契约 → 规格文件
```

测试是最准确的"活文档"。AI 从测试用例能推断出准确的接口契约、边界条件和依赖关系。

### 策略 D：有 TypeScript 类型的项目

```
.d.ts / interface / type → AI 提取类型系统 → 规格文件
```

TypeScript 项目的逆向成本最低，因为类型定义本身就是半结构化的规格。

### 策略 E：多人维护的遗留项目

```
Git blame → 找到各模块的核心贡献者 → 各自审核对应模块的规格草稿
```

多人项目的隐性知识分散在不同人脑中。需要让各模块的"活字典"审核对应的规格。

### 策略 F：准备重构 / 迁移的项目

```
先建当前状态的规格（AS-IS）→ 写目标状态的规格（TO-BE）→ 差异就是重构任务
```

最有价值的场景。逆向出的 AS-IS 规格本身就是重构计划的输入。

## ✅ 7. 逆向建规格检查清单

### 启动前
- [ ] 已安装 OpenSpec CLI（`npm install -g @fission-ai/openspec@latest`）
- [ ] 已在项目中运行 `openspec init`
- [ ] 已列出项目的核心模块清单（按依赖顺序排列）

### 每个模块
- [ ] 让 AI 读取模块的核心文件（service、store、pages）
- [ ] AI 生成规格草稿，包含：职责、技术方案、数据模型、API、对外暴露、依赖
- [ ] 你审核并修正了以下内容：
  - [ ] 技术方案的准确性
  - [ ] 隐性约定和禁用项
  - [ ] 模块间依赖关系
  - [ ] 未来规划和待办事项
- [ ] 规格文件已存入 `openspec/specs/`
- [ ] 让 AI 基于新规格做一次简单任务，验证规格的有效性

### 完成后
- [ ] 所有核心模块都有规格文件
- [ ] 有一个 `conventions.md` 记录项目级约定
- [ ] 有一个 `tech-stack.md` 声明技术栈
- [ ] 团队成员知道规格文件的存在和更新流程

## 💡 实战心得

### 1. 不要追求 100% 完美

逆向建的规格不需要覆盖每一行代码。**覆盖 80% 最核心的设计决策，就能解决 90% 的 AI 理解问题。**

### 2. specs/ 是给 AI 看的，不是给人看的

不需要写得文采飞扬。关键字段要准确、格式要统一、该列的不漏。AI 读结构化文本的效率比读散文高 10 倍。

### 3. "约定"比"实现"更重要

规格文件里最有价值的不是"用了什么"，而是"为什么用这个而不用那个"。比如：

```markdown
## 禁用项
- ❌ 不要使用 @antmjs/vantui（项目已迁移到 @taroify/core）
- ❌ 不要直接使用 fetch（使用封装好的 useRequest Hook）
- ❌ 不要使用 Redux（项目统一使用 Zustand）
```

这三行"不要做什么"，比一页"做了什么"的价值更高。

### 4. 每次改动后更新规格

建完规格只是开始。之后每次功能迭代完，用 `/opsx:archive` 更新规格。否则规格很快就会和代码脱节，退化成"另一份过时文档"。

> **逆向建规格不是终点，是存量项目接入 AI 工作流的起点。** 3 小时的投入，换来的是未来每一次 AI 开发都"像老员工一样熟悉你的项目"。
