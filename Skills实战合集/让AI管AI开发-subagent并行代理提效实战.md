# 让 AI 管 AI 开发 — subagent + dispatching-parallel-agents 并行提效实战

> 🤖 让 AI 连续写 8 个任务，到第 5 个的时候输出质量已经惨不忍睹。Superpowers 的解决方案：**让一个主代理调度多个"新鲜"的子代理**，每个只管一件事，全程满状态输出。再加上并行派发，3 倍提速。

---

## 🎯 场景与挑战

你的博客系统需要同时添加 3 个独立功能：

1. **全文搜索** — Cmd+K 呼出搜索面板，基于 PostgreSQL tsvector
2. **暗黑模式** — 跟随系统主题，支持手动切换
3. **RSS 输出** — 自动生成 `/feed.xml`

这三个功能完全独立、互不影响。但如果你让 AI 在同一个会话中串行开发——

```
第 1 个功能（搜索）：质量 ⭐⭐⭐⭐⭐    上下文清洁
第 2 个功能（暗黑模式）：质量 ⭐⭐⭐⭐   开始有噪音
第 3 个功能（RSS）：质量 ⭐⭐⭐        上下文混杂，开始跑偏
```

原因很简单：**AI 的上下文窗口是有限的**。对话越长，前面的代码讨论、修改记录就越多，AI 对当前任务的专注力下降。

---

## 🧰 Skills 组合

| Skill | 职责 | 解决什么问题 |
|:---:|:---|:---|
| 🤖 `subagent-driven-development` | 为每个任务派出全新子代理 | 解决上下文退化 |
| 🔀 `dispatching-parallel-agents` | 独立任务并行执行 | 3 倍提速 |
| ✅ `verification-before-completion` | 所有任务完成后统一验证 | 确保集成没问题 |

```
工作流：
主代理（协调者，不写代码）
  ├── 子代理 A → 搜索功能 ──┐
  ├── 子代理 B → 暗黑模式 ──┼── 并行执行
  └── 子代理 C → RSS 输出 ──┘
                              ↓
              verification（统一验证）
```

## 🛠️ 实战过程

### Step 1：主代理分析任务独立性

当你说 "帮我给博客添加搜索、暗黑模式和 RSS 输出功能" 时，主代理首先分析：

```markdown
🔀 [Parallel Analysis] 任务独立性检查：

✅ 搜索功能 — 涉及文件：SearchDialog.tsx, /api/search/route.ts, 数据库迁移
✅ 暗黑模式 — 涉及文件：ThemeProvider.tsx, CSS 变量, ThemeToggle.tsx  
✅ RSS 输出 — 涉及文件：/feed.xml/route.ts, RSS 生成器

文件交集：❌ 无交集
状态依赖：❌ 无共享状态
合适并行：✅ 

决定：3 个任务并行派发。
```

### Step 2：并行派发三个子代理

主代理不写任何代码，只负责调度。为每个子代理精心准备上下文：

**子代理 A 的任务指令：**
```markdown
你的任务：实现博客全文搜索功能。

项目上下文：
- 技术栈：Next.js 14 + PostgreSQL + Tailwind CSS
- 数据库已有 articles 表，约 150 条记录
- 现有导航：Header 组件包含 Logo 和导航链接

实施要求：
1. 创建 SearchDialog 组件（Cmd+K 呼出）
2. 创建 /api/search API 路由
3. 添加 search_vector 列和 GIN 索引
4. 遵循 TDD：先写测试后写代码
5. 不要修改 Header 组件以外的现有文件

完成后返回：实现摘要 + 测试结果 + 改动文件清单
```

三个子代理**同时启动**，各自拥有全新的上下文。


### Step 4：主代理审查与集成

三个子代理都返回后，主代理执行：

```markdown
🤖 [Integration Review]

1. 文件冲突检查：
   ✅ 子代理 A 改了 Header.tsx（加搜索图标）
   ✅ 子代理 B 改了 globals.css（加 CSS 变量）
   ✅ 子代理 C 没有改现有文件
   → ❌ 无冲突

2. 测试集成：
   Running: npm test
   ✅ Search: 5/5 passed

   ✅ RSS: 3/3 passed
   Total: 12/12 passed

3. 构建验证：
   Running: npm run build
   ✅ Build successful (3.1s)
```

## 📊 效果对比

| 维度 | 串行开发（一个 AI 做完所有） | 并行子代理 |
|:---:|:---|:---|
| 第 3 个功能质量 | ⭐⭐⭐（上下文退化） | **⭐⭐⭐⭐⭐（全新上下文）** |
| 总耗时 | ~3 小时（串行） | **~1 小时（并行）** |
| 文件冲突 | 容易误改（上下文混乱） | **主代理统一检查** |
| 测试覆盖 | 后面的功能往往忘了写测试 | **每个子代理都遵循 TDD** |
| 代码审查 | 一次性审查所有代码 | **每个任务独立审查** |

## 💡 实战心得

### 1. 并行的前提是"独立"

不是所有任务都能并行。如果两个任务需要修改同一个文件、共享同一个状态，就不能并行。主代理的第一步就是**检查独立性**。

### 2. "把信息喂给子代理"而不是"让它自己找"

子代理的上下文是全新的，它不知道你的项目长什么样。主代理要**主动把必要信息写在任务指令中**，而不是让子代理自己去探索整个代码库。

### 3. 子代理不是越多越好

3 个独立任务 → 3 个子代理，完美。但如果你把一个功能拆成 10 个子代理，每个子代理做的事太碎，集成的工作量反而更大。**一个子代理 = 一个完整的功能单元**。

### 4. 适用场景

| 场景 | 是否适合 |
|:---:|:---|
| 3+ 个独立功能同时开发 | ✅ 完美场景 |
| 多个独立 Bug 同时修复 | ✅ 用 dispatching-parallel-agents |
| 一个复杂功能拆分执行 | ✅ 用 subagent-driven-development（串行子代理） |
| 需要共享状态的任务 | ❌ 不适合并行 |
| 可以但不确定独立性 | ⚠️ 先让一个子代理做，确认无冲突再并行 |
