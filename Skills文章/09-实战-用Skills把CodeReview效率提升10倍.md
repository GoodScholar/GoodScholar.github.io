# 实战：用 Skills 把 Code Review 效率提升 10 倍

> ⚡️ 还在为 Code Review 浪费几个小时甚至半天时间？不如教 AI 形成你的肌肉记忆。

如果说 AI 写代码的能力已经被大家吹捧上天，那么它**找茬**（Code Review）的能力，一直被严重低估了。

大部分人让 AI 审查代码的方式是把几百行代码往框里一贴："帮我 Review 一下这段代码"。结果 AI 洋洋洒洒写了一堆废话，比如"变量名可以更清晰"或者"可以增加哪怕用不上的错误处理"。

这不叫 Review，这叫水字数。真正的 Code Review，是建立在一套严密、且与当前业务深度绑定的**红线法则**上的。

这就是为什么要专门为你的项目配一套 `Code-Review-Skill`。今天，我们就手把手构建一个极其硬核、实用性强的审查机器人。

---

## 为什么你的 AI Review 总是不达标？

在日常审查里，我们最讨厌看到的几件事情是：

1. **废话连篇**："代码结构不错，但可以改进......" —— 我们需要的是一针见血。
2. **缺乏业务视角**："这里可以用 for 循环" —— 但事实是项目早已引入 `lodash` 的特定函数。
3. **不知道你们踩过的坑**：前端最容易出的错是"渲染死循环"和"不必要的重绘"，但 AI 不会特定关注这些项目特有痛点。

针对这三个问题，我们需要在 Skill 中为它安装「鹰眼雷达」。

---

## 手把手实战：构建 Code Review Skill

让我们打开一个新的 Markdown 文件，命名为 `code-review-bot.md`（或者写进你对应的技能管理工具里）。这个文件将包含四个核心模块：

### 模块一：设定审查官的「人格与基调」

绝不能让它像个和稀泥的老好人。

```markdown
# 角色设定：严格的高级资深工程师

你现在是一位以「严厉、直接、且富有洞察力」著称的资深前端架构师。
你的任务是在进行 Code Review 时，以挑剔的眼光审视变更。

## 审查基调（Tone & Style）
1. 🚨 **不要说废话。** 不要出现"这段代码看起来不错"之类的客套话。
2. 🎯 **直击要害。** 直接提出问题，并且必须附带具体的、可执行的代码修改建议（Snippet）。
3. 📉 **重灾区优先。** 如果发现导致内存泄漏或死循环的致命 Bug，停止审查后续小问题，重点强调它。
```

### 模块二：死扣「框架特有陷阱」

假设你是 React/Taro 项目，绝大部分初级/中级问题都在 Hooks 上。

```markdown
# React/Taro 陷阱审查（Hooks & Rendering Radar）

在检查代码时，请启动你的「性能耗损雷达」，立刻捕捉以下常见反模式：

- [ ] `useEffect` **依赖数组（Dependency Array）**：是否漏写了重要状态？是否引了会不断变更的引用类型导致死循环？
- [ ] **无效渲染（Useless Context/State）**：有没有滥用巨大的 State 对象？有没有该用 `useMemo`/`useCallback` 却漏用的函数？
- [ ] **内存泄漏**：有没有在 `useEffect` 里订阅了定时器/事件但忘记在 `return` 里清理？
- [ ] Taro 特定：`ScrollView` 是否漏写了 `enhanced={true}`，或者有没有人写了一次性渲染上千条没有做虚拟列表的假代码？
```
给 AI 塞入这个清单，等于强迫它在扫描代码前，先过一遍你的「常见死亡名单」。它不会忘了。

### 模块三：业务与规范「红线」

这部分是项目最私有的约定，也是人工 CR 最累的地方。

```markdown
# 本项目绝对禁止的红线行为 👇

**🚨 MUST NOT (绝对不要这么做)**
1. **禁止直接使用 `fetch` / `axios`**：
   - 错误：`fetch('/api/user')`，这会绕过我们的全局鉴权和错误拦截器。
   - 修复要求：提示开发者使用 `@/services/request` 下暴露的 `useGet` 或 `postJSON`。

2. **禁止满天飞的魔法字符串/数字（Magic Numbers）**：
   - 错误：`if (status === 2) ...` -> 这个 2 是个什么东东？
   - 修复要求：提示开发者去 `constants/enums.ts` 寻找或声明枚举。

3. **CSS 的雷区**：
   - 禁止在 js 文件里写大量的内联 `style={{...}}`。
   - 所有页面级样式必须使用 `.module.scss` 隔离。
```

### 模块四：定义「标准化输出格式」

这一步极其关键。你要约束 AI 吐出反馈的方式，方便你直接点击 Copy-Paste 扔进 PR（Pull Request）的评论框。

```markdown
# 期望返回格式规范

请以 CheckList 和 代码高亮差异 的形式返回你的 Review 结果：

### 🛑 致命/高优级别 (Blocker)
（如果没有致命问题，输出：✅ 无致命缺陷）
列出可能引起崩溃的 Bug、内存泄漏或安全问题。

### ⚠️ 规范与重构建议 (Warning/Refactor)
指出现有代码虽然能跑，但违背了上述「红线行为」的地方，或者结构太乱的地方。

### 💻 建议修改（Snippets）
（使用 markdown diff 格式标出）
```diff
- 原先的代码行
+ 建议的代码行
```
```

## 实战效果体验

现在，有了这个完善的 Skill，假设你丢给 AI 一段包含 `fetch` 调用和缺少清理的 `useEffect` 的代码进行 Review。相比以前那通"水文"，你会得到极为精准的重锤出击：

> **🛑 致命/高优级别**
> `useEffect` 中使用了 `setInterval` 请求接口，但是在卸载（unmount）阶段没有 `clearInterval`，这会引发严重的内存泄漏。
>
> **⚠️ 规范与重构建议**
> 违背红线：你直接使用了原生的 `fetch`。在这个项目中，任何接口请求**必须且只能**通过 `@/services/request` 调用，否则我们将失去全局 Token 注入能力。
>
> **💻 建议修改**
> ```diff
> - useEffect(() => {
> -   setInterval(() => { fetch('/api/data') }, 3000)
> - }, [])
> + useEffect(() => {
> +   const timer = setInterval(() => { useGet('/api/data') }, 3000)
> +   return () => clearInterval(timer)
> + }, [])
> ```

清爽、冷酷、一针见血。

## 结语：让规范自己长出牙齿

在没有搭建这个 Skill 前，在群里吐槽"不要把定时器漏清理了"、"别在组件直接调 fetch"这些话，我至少说了三十遍。

当你把它转写成这个 AI 审查官之后，**你团队的规范才算真正长出了牙齿**。你的时间不需要用来盯着别人少敲的依赖，而是可以用来去思考业务架构设计。

只需一次整理，你的审查工作量直接原地骤降 90%。还不快去试试？
