# Superpowers 细节分析与实战使用指南

> 🛡️ 当你的 AI 编程助手写出了一大堆代码却没人知道对不对的时候，你需要的不是更快的代码生成，而是**工程纪律**。这正是 Jesse Vincent 创建 **Superpowers** 这一套结构化工作流技能包的核心动机——它不会替你写更多代码，而是保证每一行代码都是高质量的。

---

### 背景引入：AI 编程最大的敌人不是能力，是"放任"

很多人用 Claude Code 或 Gemini CLI 时习惯说："帮我加一个用户管理模块，要有增删改查。"

结果呢？AI 洋洋洒洒几百行代码一通输出。看起来很专业，但实际跑起来一堆空指针，测试？不存在的。需求对不对？没人验证过。

Superpowers 的哲学是：**流程纪律、系统性防线**。它像一位经验丰富的首席工程师，在你耳边不断低语——"想清楚了再动手"、"写完了跑测试没？"、"你确定这真的修好了？"

---

## 一、环境搭建与配置：让你的 AI 拥有"超能力"

Superpowers 支持多个 AI 编程平台，安装方式简洁明了：

### Claude Code（推荐）

```bash
# 方式一：官方插件市场
/plugin install superpowers@claude-plugins-official

# 方式二：社区插件市场
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace
```

### Gemini CLI

```bash
gemini extensions install https://github.com/obra/superpowers
```

### Cursor

```text
/add-plugin superpowers
```

安装完成后，你无需任何额外配置。Superpowers 的技能会**自动触发**——当 AI 检测到你要创建新功能，它会自动进入 `brainstorming`；检测到你在修 Bug，它会自动进入 `systematic-debugging`。这就是所谓的"你的 AI 助手就是有了超能力"。

## 二、七大核心技能深度拆解

Superpowers 的技能库是模块化的，每个技能都是一套严格的标准作业流程（SOP）。让我们逐一拆解：

### 1. 🧠 头脑风暴：`brainstorming`

**触发条件**：当你说"新功能"、"实现"、"添加"、"创建"等关键词时自动激活。

▪️ **核心意图**：在写任何一行代码之前，强制你和 AI 先想清楚要做什么。
▪️ **解决痛点**：开发者最常犯的错误——没想清楚就动手，写了一半发现需求理解错了。
▪️ **核心流程**：

```
探索项目上下文 → 逐个提问澄清需求 → 提出 2-3 种方案及权衡 
→ 分段展示设计 → 编写设计文档并提交 → 自审查 → 用户审查 → 交接到计划编写
```

> 💡 **亮点**：它有一条铁律——**哪怕项目再简单，也必须走设计流程。** 因为"看起来简单"的项目，恰恰是未经审视的假设最容易导致返工的地方。

▪️ **额外特性**：内置**可视化伴侣**（Visual Companion），可以在浏览器中向你展示线框图、架构图和布局方案对比，让设计讨论不再是纯文字。

---

### 2. 📋 编写计划：`writing-plans`

**触发条件**：`brainstorming` 完成后自动触发。

▪️ **核心意图**：把设计文档转化为一个傻瓜式的、一步一动的实施计划。
▪️ **解决痛点**：AI 拿到模糊需求后自由发挥，写出一坨看不懂也改不动的代码。
▪️ **核心特色**：

每个步骤都是 **2-5 分钟的原子操作**，严格遵循 TDD：
```markdown
- [ ] Step 1: 写失败的测试用例
- [ ] Step 2: 运行测试，确认失败
- [ ] Step 3: 写最小实现代码
- [ ] Step 4: 运行测试，确认通过
- [ ] Step 5: 提交代码
```

> 💡 **铁律**：**不允许任何占位符。** "TBD"、"TODO"、"后续补充"——这些在 Superpowers 的计划中是禁止出现的。每一步都必须包含完整的文件路径、具体的代码、精确的验证命令。

---

### 3. 🤖 子代理驱动开发：`subagent-driven-development`

**触发条件**：计划编写完成后用户选择执行方式时激活。

▪️ **核心意图**：这是 Superpowers **最前沿的执行引擎**，开创了"AI 管理 AI"的工作模式。
▪️ **解决痛点**：长对话中 AI 的上下文越来越混乱，导致后面的代码质量急剧下降。
▪️ **核心流程**：

```
读取计划 → 逐个任务派发"实现子代理"（全新上下文）
→ 实现完毕后派发"规格审查子代理"（检查是否符合设计）
→ 通过后派发"代码质量审查子代理"（检查代码是否优雅）
→ 全部通过 → 标记完成 → 下一个任务
```

> 💡 **为什么这么做？** 每个子代理都拥有全新的上下文——不带主会话的任何历史包袱。这意味着即使你的项目持续开发了几个小时，每个任务仍然会被一个"精力充沛"的 AI 来执行。而主代理只做协调调度，不做具体实现，从而保持自己的上下文清洁。

▪️ **智能模型选择**：简单机械任务用便宜快速的模型，复杂架构判断任务用最强模型——既省钱又保质量。

---

### 4. 🔴 测试驱动开发：`test-driven-development`

**触发条件**：任何涉及功能实现或 Bug 修复的场景。

▪️ **核心意图**：这不是"建议你写测试"，而是**铁腕执行 TDD**。
▪️ **解决痛点**："先写代码再补测试"的行为在 Superpowers 体系下被定性为**工程事故**。

▪️ **铁律**（The Iron Law）：
```
没有失败的测试 → 不允许写实现代码
先写了代码再补测试 → 删掉代码，从头来过
```

▪️ **RED-GREEN-REFACTOR 循环**：

| 阶段 | 做什么 | 验证什么 |
|------|--------|----------|
| 🔴 RED | 写一个最小的失败测试 | 确认测试因"功能缺失"而失败，而不是因为拼写错误 |
| 🟢 GREEN | 写最少的代码让测试通过 | 不多写一行——严禁过度工程 |
| 🔵 REFACTOR | 整理代码 | 保持测试始终绿色 |

> 💡 **灵魂拷问表**：Superpowers 列出了一张"借口粉碎表"——
> - "太简单不需要测试" → 简单代码也会出 bug，测试只需 30 秒。
> - "我先手动测过了" → 手工测试是随机的，无法回归。
> - "删掉几小时的工作太浪费" → 沉没成本谬误。留着没测过的代码才是真浪费。

---

### 5. 🔍 系统化调试：`systematic-debugging`

**触发条件**：当你说"bug"、"报错"、"失败"、"不工作"等关键词时自动激活。

▪️ **核心意图**：终结"我觉得可能是这里的问题，改一下试试"的赌博式开发。
▪️ **解决痛点**：随机修改浪费时间，还容易制造新的 Bug。

▪️ **铁律**（The Iron Law）：
```
没有完成根因调查 → 不允许提出修复方案
```

▪️ **四阶段调试协议**：

| 阶段 | 核心活动 | 成功标准 |
|------|----------|----------|
| **Phase 1: 根因调查** | 仔细阅读报错、稳定复现、检查最近变更、追踪数据流 | 理解了**是什么**和**为什么** |
| **Phase 2: 模式分析** | 找到类似的正常工作的代码，对比差异 | 找出差异点 |
| **Phase 3: 假设与验证** | 提出单一假设，做最小修改来验证 | 假设被确认或否定 |
| **Phase 4: 实施修复** | 先写失败测试 → 修复根因 → 验证 | Bug 解决，测试通过 |

> 💡 **3 次修复失败保险机制**：如果连续 3 次修复都失败了，Superpowers 会要求你停下来——这不是假设错了，而是**架构本身有问题**。此时必须与团队讨论架构变更，而不是继续打补丁。

▪️ **实战效果对比**：
| | 系统化调试 | 随意修改 |
|---|---|---|
| 平均修复时间 | 15-30 分钟 | 2-3 小时 |
| 一次修复成功率 | 95% | 40% |
| 引入新 Bug | 接近零 | 常有发生 |

---

### 6. ✅ 完成前验证：`verification-before-completion`

**触发条件**：当 AI 即将宣布"做完了"时自动激活。

▪️ **核心意图**：堵住 AI 最常见的"幻觉"——明明没跑过测试就说"所有测试通过"。
▪️ **解决痛点**：AI 喜欢说"应该没问题了"、"看起来是对的"这种没有依据的判断。

▪️ **铁律**（The Iron Law）：
```
没有新鲜的验证证据 → 不允许声明完成
```

▪️ **检验门函数**：

```
1. 确定：用什么命令可以证明你的声明？
2. 运行：执行完整的验证命令（新鲜的、完整的）
3. 阅读：仔细检查完整输出、退出码、失败计数
4. 核实：输出是否确认了你的声明？
5. 只有全部通过，才可以表态
```

> 💡 **红旗拦截器**：只要 AI 的输出中出现了以下任何词汇——"应该"、"可能"、"似乎"、"看起来"——系统就会自动拦截，禁止声明完成。**有证据再说话，这是不可谈判的。**

---

### 7. 🏁 收尾开发分支：`finishing-a-development-branch`

**触发条件**：所有任务完成后自动激活。

▪️ **核心意图**：为整个开发流程画上一个规范的句号。
▪️ **核心流程**：

```
验证所有测试通过 → 提供 4 个标准选项：
  1. 本地合并到主分支
  2. 推送并创建 Pull Request
  3. 保持分支现状
  4. 丢弃此次工作
→ 执行用户选择 → 清理工作区
```

> 💡 **双重保险**：即使你选择合并，Superpowers 也会在合并后再跑一次完整测试。选择丢弃时，必须输入"discard"进行确认——防止手滑误删。

## 三、最佳实践：全链路"纪律型"工作流

那么，Superpowers 的全套流程是怎么串起来的呢？

> 💡 **实战案例演绎（从零到一）**：
> 1. 你：打开终端，输入 `我想给项目加一个评论系统。`
> 2. **`brainstorming` 自动介入** → AI 逐个问你问题：支持嵌套回复吗？要实时更新吗？数据存哪里？然后提出 3 种方案供你选择。
> 3. 你确认方案后 → AI 写出设计文档，自审后请你审阅。
> 4. **`writing-plans` 自动接替** → 把设计拆成 12 个原子任务，每个 2-5 分钟，全部带 TDD 步骤。
> 5. 你确认执行 → **`subagent-driven-development` 启动** → 主代理逐个派发子代理执行任务。
> 6. 每个任务完成后自动经历**双重审查**：规格审查员确认符合设计，代码审查员确认代码质量。
> 7. 期间遇到一个诡异的 Bug → **`systematic-debugging` 介入** → 四阶段调试找到根因，一次修复。
> 8. 全部任务完成 → **`verification-before-completion` 拦截** → 强制运行完整测试套件。
> 9. 测试全绿 → **`finishing-a-development-branch` 收尾** → 你选择创建 PR，代码推送，分支清理。

整个流程：**构思 → 设计 → 计划 → 实现 → 审查 → 调试 → 验证 → 交付**，环环相扣，全程有纪律。

## 四、Superpowers vs GStack：什么时候该用谁？

读到这里，你可能会想起我们之前介绍的 GStack。两者该如何选择？

| 场景 | 推荐 | 原因 |
|------|------|------|
| 黑客马拉松，48 小时交付 MVP | **GStack** | 虚拟团队全包揽，快速交付 |
| 大型项目重构，不能出错 | **Superpowers** | 四阶段调试 + TDD + 双重审查 |
| 需要 AI 操作浏览器做 QA | **GStack** | `/qa` + `/browse` 无头浏览器 |
| 多人协作，追求代码规范 | **Superpowers** | 子代理开发 + 自动 Code Review |
| 从零快速构建并发布 | **GStack** | `/ship` 一键发版 |
| 解决疑难杂症 Bug | **Superpowers** | 系统化调试协议（四阶段） |

> 🤝 **最强组合**：在项目初期用 **GStack** 快速搭建原型，进入稳定开发后切换到 **Superpowers** 保障质量。两套技能包完全兼容，可以在同一个项目中并存！
