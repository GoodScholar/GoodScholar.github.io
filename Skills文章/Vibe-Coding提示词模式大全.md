# Vibe Coding 提示词模式大全

> 这篇文章是一本「Vibe Coding 速查手册」。收录了 12 个高频开发场景的提示词模板，每个都配有好坏对比和使用技巧。建议收藏，需要的时候翻出来看。

---

## 📖 使用说明

每个模式包含以下部分：

- **🎯 场景**：什么时候用
- **🟢 好的提示词**：推荐的写法
- **🔴 坏的提示词**：常见的错误写法
- **💡 技巧**：提升效果的小窍门

---

## 模式 1：🏗️ 项目创建

### 🎯 场景

从零开始一个新项目。

### 🔴 坏的提示词

```
帮我做一个网站
```

→ AI 不知道做什么类型的网站，不知道用什么技术，不知道要什么功能。

### 🟢 好的提示词

```
帮我创建一个个人技术博客项目：

技术选型：
- 框架：Next.js 14 (App Router)
- 样式：Tailwind CSS
- 内容管理：MDX 文件
- 部署：Vercel

核心功能（MVP）：
1. 首页展示文章列表，按时间倒序
2. 文章详情页，支持 Markdown 渲染 + 代码高亮
3. 按标签筛选文章
4. 暗色/亮色主题切换
5. 响应式设计，移动端适配

不需要（先排除）：
- 评论系统
- 搜索功能
- 多语言支持
```

### 💡 技巧

1. **明确说「不要什么」**：排除法比罗列法更有效，防止过度工程
2. **先定技术栈**：避免 AI 随便选，选到你不熟悉的
3. **标注 MVP 范围**：让 AI 聚焦，别一上来就做完整版

## 模式 2：✨ 功能添加

### 🎯 场景

给现有项目添加一个新功能。

### 🔴 坏的提示词

```
加一个搜索功能
```

### 🟢 好的提示词

```
给博客添加文章搜索功能：

需求：
- 首页顶部添加搜索框
- 输入时实时过滤文章列表（前端过滤就行，不需要后端）
- 搜索范围：文章标题 + 标签
- 搜索词高亮显示在结果中
- 无结果时显示友好提示

技术约束:
- 不要引入新的依赖库
- 保持现有的文章列表组件结构不变
- 搜索逻辑放在独立的 hook 中
```

### 💡 技巧

1. **说清楚「不要引入新依赖」**：否则 AI 可能装一堆你不需要的库
2. **说清楚「保持现有结构」**：避免 AI 大改你的代码
3. **定义行为细节**：实时过滤还是点按钮搜索？前端还是后端？

## 模式 3：🐛 Bug 修复

### 🎯 场景

代码出 Bug 了，需要 AI 帮忙找原因和修复。

### 🔴 坏的提示词

```
代码不工作了，帮我看看
```

### 🟢 好的提示词

```
遇到一个 Bug，需要排查：

现象：
- 点击「提交」按钮后页面卡死，没有任何反应
- 控制台没有报错
- 在 Chrome 正常，Safari 里出问题

预期行为：
- 点击后应该发送 POST 请求到 /api/submit
- 请求成功后跳转到 /success 页面

已经尝试过的：
- 在 onClick 里加了 console.log，确认事件触发了
- 网络面板看到请求发出去了，但一直 pending

相关代码在这几个文件中：
- src/components/SubmitForm.tsx
- src/hooks/useSubmit.ts
- src/api/submit.ts
```

### 💡 技巧

1. **描述现象而不是猜测原因**：让 AI 自己推理
2. **说「已经试过什么」**：避免 AI 建议你已经做过的事
3. **指出相关文件**：缩小排查范围

## 模式 4：♻️ 代码重构

### 🎯 场景

代码能跑，但结构不好，想让 AI 帮你重构。

### 🔴 坏的提示词

```
把这段代码重构一下
```

### 🟢 好的提示词

```
重构 src/utils/dataProcessor.ts，目标：

当前问题：
1. 一个函数 500 行，做了太多事
2. 多处重复的数据转换逻辑
3. 没有错误处理，try-catch 缺失
4. 变量命名不清晰（a, b, temp 之类）

重构方向：
- 拆分为 3-4 个职责清晰的函数
- 提取公共的数据转换为独立工具函数
- 添加输入校验和错误处理
- 改善命名，遵循项目现有的命名风格

约束：
- 不改变对外暴露的接口（函数签名和返回值不变）
- 不引入新依赖
- 重构后跑一遍现有测试确保不破坏功能
```

### 💡 技巧

1. **「不改接口」是最重要的约束**：避免重构变成重写
2. **说清楚「为什么」要重构**：AI 才能对症下药
3. **要求跑测试**：重构最怕引入回归 Bug

## 模式 5：🎨 UI 设计

### 🎯 场景

需要 AI 帮你做 UI 界面。

### 🔴 坏的提示词

```
做一个好看的页面
```

### 🟢 好的提示词

```
设计一个设置页面，风格参考 Linear 和 Notion：

整体风格：
- 暗色主题 (#0f1117 背景)
- 左侧导航 + 右侧内容区的布局
- 字体用 Inter，中文用系统默认
- 卡片使用微妙的玻璃态效果（backdrop-filter: blur）

左侧导航：
- 分组：个人信息、通知设置、外观设置、安全设置
- 当前选中项有蓝色高亮条
- 导航宽度 240px，可折叠

右侧内容：
- 每个设置项一行，左边标签 + 说明文字，右边是操作控件
- Toggle 开关、Select 下拉框、Input 输入框混合使用
- 修改后底部浮出「保存修改」按钮

动效：
- 切换导航项有平滑过渡（0.2s ease）
- 保存按钮从底部滑入
- Toggle 切换有弹性动画
```

### 💡 技巧

1. **给参考案例**：「参考 Linear」比「做得好看」有用 100 倍
2. **给具体色值**：别说「暗色」，说 `#0f1117`
3. **描述动效**：动效是区分「凑合」和「精致」的关键

## 模式 6：📝 文档生成

### 🎯 场景

为项目或 API 生成文档。

### 🟢 好的提示词

```
为 src/api/ 目录下的所有 API 接口生成文档：

格式要求：
- Markdown 格式
- 每个接口包含：请求方法、路径、参数说明、返回值说明、示例请求和响应
- 参数表格包含：字段名、类型、是否必填、说明
- 按模块分组（用户模块、文章模块、设置模块）

风格要求：
- 简洁，不要废话
- 示例中使用有意义的假数据（不要 foo/bar）
- 错误响应也要列出来
```

## 模式 7：🧪 测试编写

### 🟢 好的提示词

```
为 src/hooks/useAuth.ts 编写单元测试：

测试框架：Vitest + React Testing Library

覆盖场景：
1. 正常登录流程（输入正确凭证 → 获取 token → 跳转首页）
2. 登录失败（错误密码 → 显示错误提示 → 不跳转）
3. 未登录状态访问保护路由 → 重定向到登录页
4. Token 过期 → 自动刷新 → 刷新失败则退出登录
5. 并发请求时的 Token 刷新（只触发一次刷新）

Mock 要求：
- Mock fetch 请求
- Mock 路由跳转
- Mock localStorage 操作

每个测试用 describe/it 组织，用 AAA 模式（Arrange-Act-Assert）
```

## 模式 8：🔄 API 集成

### 🟢 好的提示词

```
集成 Stripe 支付 API：

需求：
1. 创建一次性支付（Product 模式）
2. 支持微信支付和支付宝
3. 支付成功后回调更新订单状态

技术要求：
- 后端用 Node.js + Express
- Stripe API Key 通过环境变量管理
- Webhook 签名验证
- 支付状态用枚举管理（pending → processing → success → failed）
- 所有 Stripe 操作有错误处理和日志

安全要求：
- Webhook endpoint 验证 Stripe 签名
- 不在客户端暴露 Secret Key
- 支付金额在服务端计算（不信任客户端传来的金额）
```

## 模式 9：📊 数据库设计

### 🟢 好的提示词

```
设计一个电商系统的数据库 Schema：

核心实体：
- 用户（User）：头像、昵称、手机号、注册时间
- 商品（Product）：标题、描述、价格、库存、分类、图片列表
- 订单（Order）：用户、商品、数量、总价、状态、创建时间
- 评论（Review）：用户、商品、评分、内容、图片

业务规则：
- 一个用户可以有多个订单
- 一个订单只包含一种商品（简化 MVP）
- 订单状态：待支付 → 已支付 → 已发货 → 已完成 → 已取消
- 每个用户对每个商品只能评论一次

技术选型：PostgreSQL
输出格式：Prisma Schema

请同时生成必要的索引建议。
```

## 模式 10：⚡ 性能优化

### 🟢 好的提示词

```
优化首页的加载性能：

当前问题：
- Lighthouse 性能分数：45 分
- 首屏加载时间：4.2 秒
- 最大内容绘制 (LCP)：3.8 秒
- 累积布局偏移 (CLS)：0.25
- JS Bundle 大小：1.2MB

已知瓶颈：
1. 首页加载了全部文章数据（200+ 篇）
2. 图片没有做懒加载
3. 没有代码分割，所有路由打包在一起

期望目标：
- Lighthouse 分数 > 85
- 首屏 < 2 秒
- Bundle < 300KB（首屏）

约束：
- 不改变现有的功能和用户体验
- 优先做投入产出比高的优化
```

## 模式 11：🔐 安全审查

### 🟢 好的提示词

```
以安全工程师的视角审查以下代码：

审查范围：src/api/ 和 src/middleware/ 目录

重点检查：
1. 鉴权和授权漏洞（是否有未保护的接口）
2. 注入攻击（SQL 注入、XSS、命令注入）
3. 敏感数据处理（密码存储、Token 管理、日志脱敏）
4. CORS 配置是否过于宽松
5. 速率限制是否缺失
6. 文件上传的安全验证

输出格式：
按严重等级分类（🔴 高危 / 🟡 中危 / 🟢 低危）
每个问题包含：位置、描述、修复建议、修复代码
```

## 模式 12：📦 项目部署

### 🟢 好的提示词

```
帮我配置这个 Next.js 项目在 Vercel 上的部署：

需要：
1. 生产环境的环境变量配置清单
2. vercel.json 配置文件（包含重定向规则和 Headers）
3. 构建脚本优化（确保构建时做类型检查和 lint）
4. 自定义域名的 DNS 配置说明

当前情况：
- 本地 npm run build 可以成功
- 使用了 next/image，需要配置远程图片域名白名单
- 有 3 个 API Route（/api/auth, /api/posts, /api/comments）
- 环境变量有：DATABASE_URL, NEXTAUTH_SECRET, GITHUB_ID, GITHUB_SECRET

安全提醒：
- 不要把真实密钥写在配置文件里
- 使用 Vercel 的 Environment Variables 面板管理
```

## 🧠 通用提示词技巧

### 1. 先描述上下文

```
项目背景：这是一个用 Vue 3 + Vite 做的内部管理后台...
当前阶段：MVP 已做完，现在在做性能优化...
团队情况：3 个前端开发者，都不太熟悉 WebSocket...
```

### 2. 明确约束条件

```
约束：
- 不引入新依赖
- 兼容 Chrome 80+
- 保持现有 API 不变
- 完成后跑一遍 npm test
```

### 3. 定义验收标准

```
做完后我会检查：
1. 首屏加载 < 2 秒
2. 所有测试通过
3. 没有 TypeScript 报错
4. Lighthouse 性能分 > 80
```

### 4. 分步迭代

```
分三步做：
1. 先搭建基础结构，我确认没问题后
2. 再加上业务逻辑
3. 最后做样式美化

每一步完成后停下来等我确认。
```

---

## 💡 最后一条建议

> **好提示词的本质不是「更长」，而是「更精准」。**

如果你发现自己写了一大段但 AI 还是理解不了，多半不是描述不够——而是你自己还没想清楚。

**把提示词当作写给自己的需求文档。如果你写完了自己都觉得很清楚，AI 大概率也能懂。**
