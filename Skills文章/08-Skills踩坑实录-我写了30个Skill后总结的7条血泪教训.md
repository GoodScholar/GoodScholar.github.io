# Skills 踩坑实录 —— 我写了 30 个 Skill 后总结的 7 条血泪教训

> ⚠️ 如果你发现自己写了长篇大论的 Skill，但 AI 依然"屡教不改"，那这篇文章就是为你准备的。

当大家第一次发现可以通过编写 Skill（或 `.cursorrules` / Custom Instructions）来"永久"调教 AI 时，通常都会陷入一种兴奋状态。于是开始把脑子里关于代码架构、命名规范、最佳实践的所有想法，一股脑地塞进一个 Markdown 文件里。

结果呢？**AI 根本不听。** 它依然会使用废弃的 API，依然会忽略你的命名规范。

在过去几个月里，我亲手写了超过 30 个不同用途的 Skills，从通用规范到特定业务模块的工作流。在这个过程中，我踩了无数的坑。

今天，我把这些踩坑经验提纯为 7 条「血泪教训」。看完这些，你的 Skill 听话程度至少能提升 300%。

---

## 教训一：不要把 Skill 写成"许愿池"
**❌ 错误示例：**
> "生成的代码要优雅、可读性强、性能好，并且要有良好的架构设计。"

很多人以为模型能理解"优雅"，但这其实是在许愿。大语言模型处理的是具体的 Token 和逻辑节点，"优雅"这个词对它来说权重极低且毫无操作性。

**✅ 正确解法：把主观形容词翻译为客观约束。**
> "代码必须满足以下约束：
> 1. 单个函数不得超过 50 行
> 2. 状态变更必须使用 Zustand 的 `set`，禁止直接操作对象
> 3. 每个导出的组件必须有 JSDoc 注释说明入参"

**要诀：如果一个规则无法用代码静态扫描工具（如 ESLint）去衡量，AI 也大概率无法严格执行它。**

---

## 教训二："MUST NOT" 比 "SHOULD" 管用一百倍
**❌ 错误示例：**
> "在使用网络请求时，你应该尽量使用我们封装的 `useRequest` hook。"

在 AI 的注意力机制里，"尽量 (should / prefer)" 是一个被弱化的软性约束。当它发现原生 `fetch` 写起来更顺手（在它的训练集里出现得更多）时，它会立刻抛弃你的"尽量"。

**✅ 正确解法：用强烈的否定句式建立绝对红线。**
> "🚨 **绝对禁止（MUST NOT/NEVER）** 在组件内直接使用 `fetch` 或 `axios`！
> **必须且只能（MUST/ONLY）** 使用 `@/hooks/useRequest` 发起请求。"

**要诀：AI 对否定词极其敏感。把防呆设计做到极致，明确指出"什么绝对不能做"比"推荐怎么做"效果好得多。**

---

## 教训三：没有正反例（Few-Shot）的规约都是纸老虎
**❌ 错误示例：**
> "请按照 BEM 命名规范来写 CSS 类名。"

只给一条规则，AI 有 50% 的几率会按它的理解胡乱生成（比如把 `block__element--modifier` 写成 `block-element_modifier`）。

**✅ 正确解法：用正反例（Anti-pattern vs Best Practice）定死格式。**
> "CSS 类名必须严格遵循 BEM 规范：
> - ❌ 错误（Anti-pattern）: `<div className="user_card active">`
> - ✅ 正确（Best Practice）: `<div className="user-card user-card--active">`"

**要诀：大模型是"靠例子（Few-Shot）"学习的神。一个好例子胜过十行文字解释。**

---

## 教训四：全能型 Skill = 无能型 Skill
**❌ 错误示例：**
> （在一个名为 `frontend-rules.md` 的文件里，塞进去 5000 字，包含了 React 规范、CSS 规范、CI/CD 提交流程、Git 命名、甚至是测试用例的写法）

这是新手最容易犯的错误——把所有的上下文压缩在一个巨无霸文件里。当任务只涉及改一点 CSS 时，AI 的上下文窗口被大量的 CI/CD 规则污染，导致指令稀释（Dilution Effect）。

**✅ 正确解法：拆分细粒度的专属 Skill。**
> - `React-Rules.md`：只管组件和状态。
> - `Styling-Rules.md`：只管 CSS 和动画。
> - `Testing-Rules.md`：只管 Jest 和 RTL。

在支持动态加载上下文的工具（如 Superpowers/GStack）中，只在相关任务触发时才挂载对应的 Skill。

**要诀：给 AI 的上下文要像手术刀一样精准，不要给它看与当前任务无关的任何东西。**

---

## 教训五：忘记声明技术的"特定版本"
**❌ 错误示例：**
> "本项目使用 React Router 管理路由。"

React Router v5 和 v6，以及新的 v7（与 Remix 融合），API 截然不同。如果你不写版本号，AI 会大概率根据它的默认训练集偏好（通常是 v5 或老版 v6）给你写出一堆废弃的 `useHistory` 代码。

**✅ 正确解法：精确到破坏性更新的版本。**
> "本项目使用 **React Router v6.4+** 引入的 Data API，必须使用 `createBrowserRouter` 和 `Loader`，绝对禁止使用 `Switch` 和 `useHistory`。"

---

## 教训六：逻辑层级混乱，不善用 Markdown 结构
AI 解析 Markdown 的能力极强，标题（H1/H2）、粗体、列表、引用栏，在解析器眼里都有**层级和权重**的差异。

如果你把所有的规则写成一堆没有排版的纯文字段落，重点会被过滤掉。

**✅ 正确解法：把 Skill 当作结构化的数据库去排版。**
1. 用 `#` 和 `##` 划分清晰的逻辑块。
2. 核心红线用 `> ⚠️ WARNING:` 引用块包裹。
3. 关键词用粗体 `**绝对约束**` 强调。
4. 使用 Markdown 表格映射旧 API 到新 API。

---

## 教训七：写完之后从不复盘（这是最大的隐患）
写完 Skill 不是终点。
当 AI 第 3 次犯了同样的错误时，绝大部分人的反应是："这 AI 怎么这么蠢，我都写了它也不看。"

**错！大错特错。**
如果 AI 持续犯同一个错误，唯一的可能就是 **你的这段 Skill 描述有歧义，或者约束力不够。**

你需要立刻去复盘并修改 Skill。加上更严厉的词（NEVER），增加一个明显的负面例子（Anti-pattern），或者给它加上高亮（警告标志）。

**要诀：写 Skill 本身就是一个调试（Debug）过程。你不是在写文档，你是在为 AI 编写控制参数。**

---

## 总结

编写一条优秀的 Skill 规则，应该像是在写**防御性代码**：预判 AI 可能瞎编的逻辑、封死它"自由发挥"的空间、给出可复制的标准模板、并且永远用不可辩驳的机器语态发号施令。

当你把"写长篇大论"转变为"建立精确约束"时，你会发现你的 AI 从一个"听不懂人话的实习生"，瞬间变成了"严丝合缝执行军规的特种兵"。

这，就是掌握 AI 编程操作系统的第一步。
