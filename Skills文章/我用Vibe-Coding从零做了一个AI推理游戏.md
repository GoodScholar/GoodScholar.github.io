# 我用 Vibe Coding 从零做了一个 AI 推理游戏

> 上一篇文章讲了 Vibe Coding 的概念。概念再好，不如拿项目说话。这篇文章记录我用 Vibe Coding 从**一个模糊的想法**到**一个可以玩的 AI 推理游戏**的全过程——包括哪些地方 AI 做得超出预期，哪些地方我不得不亲自下场。

---

## 🎮 项目介绍：迷局 AI 是什么

「迷局 AI」是一个 AI 驱动的推理解谜游戏。玩家扮演侦探，在一个悬疑场景中：

- 🗣️ 与 AI 驱动的 NPC 角色对话，套取线索
- 🔍 在场景中搜索物证，拼凑真相
- 🧩 根据收集的线索做出推理判断
- ⚖️ 最终指出真凶并给出推理依据

每个 NPC 都有独立的性格、动机和秘密，他们的回答会根据你的提问方式动态变化——因为背后是真正的大语言模型在驱动对话。

**技术栈**：Vite + Vanilla JavaScript + Gemini API

---

## 🛤️ 从零到一的过程

### 第一步：用自然语言描述游戏概念

我没有画原型图，没有写技术方案文档。我打开 AI 编程助手，直接说：

```
我想做一个 AI 驱动的侦探推理游戏。
玩家在一个场景（比如午夜画廊）里调查一起案件，
可以和多个 NPC 对话，每个 NPC 由大语言模型驱动，有自己的性格和秘密。
玩家要通过对话和搜证来推理出真相。

技术栈用 Vite + Vanilla JS，保持简单。
用 Gemini API 做 LLM 交互。
```

**AI 做了什么**：
- 分析了游戏的核心机制
- 提出了项目架构方案
- 识别了关键模块：场景系统、NPC 对话系统、线索收集系统、推理判定系统
- 给出了文件目录结构建议

**我做了什么决策**：
- 确认用 Vite + Vanilla JS（不用 React，追求简单）
- 选择 Gemini 作为 LLM 后端（免费额度够用）
- 先做「午夜画廊」一个场景作为 MVP

### 第二步：让 AI 搭建项目骨架

```
好，按这个方案，帮我从零搭建项目。
Vite 项目，Vanilla JS，创建好目录结构和基础配置文件。
```

**5 分钟后**：
- ✅ Vite 项目初始化完成
- ✅ 目录结构创建完成（`src/modules/`、`src/data/`、`src/styles/`）
- ✅ 基础样式和入口文件准备好
- ✅ `.env` 配置和 Gemini API 接入代码写好

**传统方式需要多久**：半天到一天（查文档、搭脚手架、配 API、调通基础流程）。

### 第三步：逐步对话迭代，完善游戏机制

这是 Vibe Coding 最精彩的部分。我不是一次把所有需求都说完，而是**一轮一轮对话**，像是在跟一个开发团队不断讨论：

```
第 1 轮对话 → NPC 对话系统
"先做NPC对话功能。玩家选择一个嫌疑人，可以自由输入问题，
NPC 根据自己的性格和秘密来回答。"

第 2 轮对话 → 线索系统
"对话中如果NPC泄露了关键信息，要自动提取为线索，
显示在玩家的线索板上。"

第 3 轮对话 → 推理系统
"玩家收集够线索后，可以进入推理阶段，
选择凶手并提交推理依据，AI 来判定对不对。"

第 4 轮对话 → UI 和场景氛围
"整体 UI 要有侦探悬疑的氛围，暗色调，
NPC 对话像审讯室的感觉。"

第 5 轮对话 → 细节打磨
"加一个打字机效果的文字显示，
NPC 回复时先显示'正在思考...'的动画。"
```

每一轮对话后我做的事情就是：**跑起来看看 → 觉得差哪里 → 继续说**。

这就是 Vibe 的感觉——你在**跟着感觉走**，而不是在**照着文档写**。

## ⚖️ 关键决策时刻：信任 AI 还是亲自下场？

Vibe Coding 不是完全放手。在整个开发过程中，有几个关键时刻我选择介入：

### 🟢 选择信任 AI 的场景

| 场景 | 原因 |
|------|------|
| UI 布局和样式 | AI 对 CSS 的掌握已经很好，产出的视觉效果符合预期 |
| Gemini API 调用封装 | 标准 HTTP 接口调用，AI 生成的代码基本无误 |
| NPC Prompt 模板 | AI 对 Prompt 工程理解深入，生成的系统提示比我自己写的还好 |
| 游戏流程状态管理 | 逻辑清晰，AI 的状态机实现足够好 |

### 🔴 选择人工介入的场景

| 场景 | 原因 | 我做了什么 |
|------|------|-----------|
| API 密钥安全性 | AI 曾把 API KEY 硬编码在前端代码中 | 改为 `.env` + 环境变量注入 |
| API 频率限制处理 | 免费 Gemini API 有每分钟请求限制 | 设计了自动重试 + 指数退避策略 |
| NPC 人设一致性 | AI 生成的 NPC 偶尔会「跳出角色」 | 调整了系统 Prompt 结构，增加了人设约束 |
| 多场景扩展架构 | AI 的第一版设计耦合度太高 | 重新设计了场景配置分离的架构 |
| 最终推理判断逻辑 | 涉及游戏核心公平性，不能有 Bug | 亲自审阅并修改了判定算法 |

### 💡 总结的经验法则

> **80/20 原则在 Vibe Coding 中特别适用**：
> - 80% 的代码可以放心交给 AI
> - 20% 涉及安全、核心逻辑、架构决策的部分需要人工把关

## 🤔 什么样的项目适合 Vibe Coding？

基于这次实践经验，我总结了一个判断框架：

### ✅ 非常适合

- **个人项目 / 独立开发**：不需要多人协作的代码规范讨论
- **原型验证 / MVP**：快速验证一个想法可不可行
- **前端密集型**：UI、样式、交互动效——AI 的强项
- **API 调用整合**：标准的 HTTP 通信、数据处理
- **工具类应用**：效率工具、数据可视化、自动化脚本

### ⚠️ 谨慎使用

- **金融 / 医疗系统**：安全性和准确性要求极高
- **高并发后端**：性能优化需要深入的领域知识
- **跨团队协作大型项目**：需要严格的代码规范和 Review 流程

### ❌ 不太适合

- **底层系统编程**：操作系统、编译器、数据库引擎
- **已有大型遗留系统的改造**：上下文过于复杂
- **有严格合规要求的系统**：每行代码都需要可追溯

## 💡 给想尝试 Vibe Coding 的你

### 5 条实用建议

1. **从小项目开始**
   不要上来就 Vibe 一个大系统。先做一个周末项目，感受节奏。

2. **学会描述意图，而不是描述代码**
   ❌ "用 addEventListener 绑定一个 click 事件"
   ✅ "点击这个按钮后弹出确认对话框"

3. **保持对话式迭代**
   不要试图一次描述完所有需求。一步步来，看到效果再决定下一步。

4. **知道什么时候该看代码**
   涉及安全性、核心逻辑、数据处理的代码——一定要看。样式、布局、标准流程——可以信任 AI。

5. **使用 Skills 提升一致性**
   给你的 AI 助手装上 Skills（比如 Superpowers），让它按结构化流程工作，而不是每次都重新发挥。
