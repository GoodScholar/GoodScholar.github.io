# Skills 组合术 —— 让 5 个 Skills 产生 50 个功能的化学反应

> 🔗 如果你还在为了完成一个新需求，给 AI 写几百字的"小作文"，你还没有领略到工程化 AI 的真正威力。真正的提效，是让系统自我运转。

当大部分人刚刚学会写「单一职责」的 Skill 时，他们通常会面临一个新的瓶颈：**"既然不要把所有规则写到一起，那我面对复杂需求时，是不是要一个个翻着文件跟 AI 确认？"**

这就是为什么我们需要**「组合术」**。

就像 Unix 的设计哲学——把每个工具做得极其单一（`ls`, `grep`, `awk`），然后用管道符（`|`）把它们串联起来，就能完成任何复杂的数据处理。

在 AI 编程中，这也是一种最顶级的范式：**将原子级的 Skills 通过架构、约束、或 Workflows 组合起来，形成一条全自动的工业流水线。**

---

## 为什么「巨无霸 Prompt」必然失败？

在进入组合术前，我们要先明白为什么大家喜欢写上千字的"巨无霸 Prompt"。

假设你要让 AI 做一个"带有权限控制和数据校验的后台用户列表页"：
- 里面包含 **React 组件风格** 的要求（如用什么框架）
- 包含 **请求 API** 的要求（如 Axios 的拦截器如何生效）
- 包含 **鉴权设计** 的要求（如 Admin 能否编辑，Guest 能否查看）
- 包含 **样式和 CSS** 的要求（如用 BEM 还是 Tailwind）

把这几部分拼在一起就是一篇"小作文"。但 AI 会晕——在它巨大的上下文矩阵里，CSS 规则很容易污染关于数据校验逻辑的注意力权重。结果往往是写对了一部分、写飞了另一部分。这就是 **"注意力摊薄效应（Token Dilution）"**。

---

## 第一阶：原子化解构（拆散）

打破这种窘境的第一步，就是把这些臃肿的指令拆分成互不干扰的「原子 Skill」。

这里我列举 5 个非常经典的原子 Skill：

1. **`Skill-React`**（纯管 React：使用箭头函数、`React.memo`）
2. **`Skill-UI库`**（纯管样式：只用 `@taroify/core`）
3. **`Skill-API`**（纯管请求层：用 `useRequest` hook、如何接后端返回的泛型结构）
4. **`Skill-RBAC`**（纯管鉴权：必须包裹 `AuthGuard` 高阶组件）
5. **`Skill-Form`**（纯管表单模型校验规则）

这些细粒度文件的存在，让每个文件内部都在强调自己领域的红线：**“不用 `auth.role === 'ADMIN'` 而是用我们专用的 `<Permission roles={['admin']}>` 包装器！”**。极其聚焦。

---

## 第二阶：按场景动态「拼装」

你当然没有必要每次干活前把这 5 个词条全过一遍。你需要的是「依赖注入」和「场景匹配」。

### 场景 A：仅仅写一个纯UI按钮（Button.tsx）

你要告诉你的助手加载哪些技能？
`Skill-React` + `Skill-UI库`。仅此而已。
> "嘿，加载基础组件和 UI 规范，帮我封装一个具有 Loading 状态的主按钮。"（API、表单和鉴权的指令根本不需要占用你的 Token 和 AI 的脑容量。）

### 场景 B：写一个单纯对接后端的 API Hook（useUserList.ts）

加载哪些？
`Skill-API`。
> "看下后端给的 User 类型的 Swagger，基于我们的 API 封装文件生成一个发起列表请求和分页状态管理的 hook。"（不用去管 CSS 怎么写这种废话，只管死磕类型和并发逻辑。）

### 场景 C：复杂场景 —— 用户编辑表单页（EditUserPage.tsx）

这时候，我们要加载：
`Skill-React` + `Skill-UI库` + `Skill-Form` + `Skill-API` + `Skill-RBAC`。

这是真正的王炸。你会发现，一旦在多上下文注入下，AI 写出的表单页面根本不需要你重构。因为它不仅知道怎么画 UI，还知道在这个提交按钮外面包裹 `Permission` 控制，并在表单触发 `onSubmit` 后使用标准 `useRequest` 将表单状态发送出去，顺便还能处理 403 的报错。

**5 个原子的排列组合，能够应对前端开发中 50 种以上大大小小的完全不同类型的碎片任务。这就是化学反应。**

## 第三阶：终极形态 —— 以 Workflow 驱动流水线

光有组合还不足以称为"工业化"。

当我们拥有了强大的兵工厂零件（Skills）后，如果还需要人类手动指挥"挂载A，挂载B，运行C"，依然是有体力劳动的。

更高维度的解法是**写一条 Workflow 指令，让它自己调用 Skills 的组合编排**。

比如，在你的配置文件里写下一个宏大的任务定义：
**`Workflow: 新建一个标准 CRUD 后台页面`**

```markdown
# [Workflow 宏指令]
当你接到新建某个后台模块CRUD任务时，请必须严格按以下步骤分步执行。每完成一步必须暂停等我输入 "继续" （防止幻觉跑偏和 token 截断）：

### Step 1: 领域建模 (Model)
- 自动加载 `Skill-TypeScript规范`。分析字段类型，建立对应的 interface 存入 `/types`。
- 【停顿确认】等待我回复。

### Step 2: 接口封装 (Services)
- 基于刚才的模型，加载 `Skill-API请求规范`，在 `/services` 创建完整的增删改查 hook。
- 【停顿确认】等待我回复。

### Step 3: UI 切图组装 (Views)
- 挂载 `Skill-React规范` 和 `Skill-Taroify库`。调用上述封装的 hooks，构建包含了列表渲染和编辑权限的完整 `Table & Drawer` 页面。
```

看到了么？在人类视角里，你只是输入了一句：
`“新建一个商品管理的 CRUD”`。

但在 AI 系统里，这个单一词条触发了一级导弹分离指令（Workflow），而这个指令像乐高组装图纸一样，在不同的层级自动抽调出你沉淀的最硬核的原子模型（Skills），层层叠加组装。

## 结语：从「敲代码的人」到「设计流水线的人」

很多人害怕被 AI 替代。但当你掌握了 Skills 的解构和编排艺术之后，你对自己的认知会发生变化。

以前的我们是在用手搓零件（手写冗长的请求封装、复制粘贴校验逻辑）。
后来的我们在教机器人搓零件（写了一篇巨长无比还老容易失效的 Prompt）。

而现在通过 **多层 Skills 系统 + Workflow**，你终于不再是一个产线上的工人了。

**你成了这套智能产线的架构师与厂长。**

5 个底层 Skill 通过组合配置，可以横生出多达 50 种开发流水线方案。而你的任务，就是把图纸画好，并喝杯咖啡监控流水线上最终的组装成果。

别再写千字小作文给 AI 了喂，赶紧去搭你的兵工厂流水线。
