# 不看代码行不行？Vibe Coding 的质量红线

> Karpathy 说「I don't even look at the code」。你也敢吗？这篇文章不是要吓你回到逐行审阅的老路上，而是要帮你建立一套**最小化人工介入、最大化质量保障**的实用方法论。

---

## 🔥 灵魂拷问

让我们先直面这个问题：

> **Vibe Coding 出来的代码，靠谱吗？**

短答案：**大部分时候靠谱，但「大部分」两个字就是问题所在。**

一个 90% 正确的代码库意味着每 10 行代码里就有 1 行可能出问题。在一个 3000 行的项目中，那就是 300 行潜在问题代码。这些问题可能是：

- 一个未处理的边界条件
- 一个硬编码的 API 密钥
- 一个在高并发下会崩的数据库查询
- 一个看起来没问题但其实有 XSS 漏洞的输入处理

所以真正的问题不是「要不要看代码」，而是——

> **该看哪些代码、怎么看、看到什么粒度？**

---

## 💥 常见翻车现场

### 场景 1：安全漏洞

```javascript
// ❌ AI 生成的代码——直接在前端暴露了 API 密钥
const API_KEY = "sk-live-abc123xyz";
const response = await fetch(`https://api.example.com/data?key=${API_KEY}`);
```

AI 经常为了让代码「能跑」而走捷径。在它的训练数据中，很多教程和示例就是这样写的——快速、直接、但在生产环境中是灾难。

### 场景 2：性能陷阱

```javascript
// ❌ AI 生成的 NPC 对话加载——每次对话都重新请求所有历史记录
async function loadConversation(npcId) {
  const allMessages = await fetch(`/api/messages?npc=${npcId}`);
  // 没有分页，没有缓存，随着对话增多，越来越慢
  return allMessages.json();
}
```

AI 生成的代码通常是「功能正确」但「性能天真」的。它倾向于用最直觉的方式实现功能，而不考虑数据量增长后的性能影响。

### 场景 3：过度工程

```
你：帮我做一个简单的待办事项
AI：好的，我来用 Redux Toolkit + RTK Query + React Router + 
    Zod Schema Validation + Custom Hook Architecture + 
    Compound Component Pattern...
```

AI 有时候会用炮打蚊子——用企业级架构做个人小项目。结果是代码能跑，但复杂度远超需求，后续维护成本极高。

### 场景 4：NPC 跳出角色

这是我在做迷局 AI 游戏时亲身经历的：

```
玩家：你昨晚在哪里？
NPC（应该是嫌疑人，有事要隐瞒）：
  "作为一个 AI 语言模型，我没有物理存在...
   但根据剧本设定，我的角色昨晚在画廊..."
```

AI 生成的 Prompt 如果约束不够强，NPC 会「跳出角色」，破坏整个游戏体验。

## 🛡️ 质量防线体系

面对这些问题，我总结了一套「三道防线」的质量保障体系：

### 第一道防线：AI 自检

**让 AI 检查 AI 生成的代码。** 这听起来像「左手查右手」，但实际上非常有效——因为你可以用不同的 Prompt 视角让 AI 审查自己的产出。

**方式 1：自动生成单元测试**

```
每次生成功能代码后，追加一句：
"为刚才的代码写单元测试，覆盖正常路径、边界条件和异常情况。"
```

**方式 2：Code Review Agent**

```
"以高级安全工程师的视角审查以下代码，着重检查：
1. 安全漏洞（XSS、CSRF、SQL 注入、密钥泄露）
2. 性能瓶颈（N+1 查询、内存泄漏、无限循环）
3. 错误处理（未捕获的异常、缺失的错误边界）"
```

**方式 3：让 AI 质疑自己**

```
"回顾你刚才生成的所有代码。
如果你是一个恶意用户，你会怎么攻击这些代码？
列出至少 5 个潜在的攻击向量。"
```

> 💡 **关键洞察**：AI 生成代码时容易「偏向正常路径」，但当你要求它扮演攻击者/审查者时，它反而能发现自己的盲区。

### 第二道防线：人工抽检

**不是所有代码都要看，但关键代码必须看。**

#### 哪些代码一定要人工审查？

| 优先级 | 代码类型 | 原因 |
|--------|---------|------|
| 🔴 最高 | 鉴权和授权逻辑 | 安全性命门 |
| 🔴 最高 | 数据处理和存储 | 数据安全和完整性 |
| 🔴 最高 | API 密钥和配置 | 泄露 = 灾难 |
| 🟡 中等 | 核心业务逻辑 | 功能正确性 |
| 🟡 中等 | 外部服务集成 | 错误处理和超时 |
| 🟢 可跳过 | UI 布局和样式 | 视觉效果即可验证 |
| 🟢 可跳过 | 标准 CRUD 操作 | 模式化代码，AI 很少出错 |
| 🟢 可跳过 | 配置文件和脚本 | 跑一下就知道对不对 |

#### 高效抽检策略

```
Step 1: 跑起来
  → 功能层面看看对不对

Step 2: 搜索关键词
  → 在代码中搜 "password"、"secret"、"key"、"admin"、"TODO"
  → 这些地方必须人工确认

Step 3: 看入口
  → 检查 API 路由定义（谁能访问什么）
  → 检查数据库 Schema（数据结构对不对）

Step 4: 看异常路径
  → 试一些异常输入，看会不会崩溃
  → 断网、慢网络情况下的行为
```

### 第三道防线：架构约束

**这是最重要也最容易被忽略的防线——不是在生成后检查，而是在生成前约束。**

#### 使用 Skills 做架构约束

```markdown
# SKILL.md 示例：安全编码规范

## 绝对禁止
- ❌ 绝对不在前端代码中硬编码 API 密钥
- ❌ 绝对不用 eval() 和 innerHTML 处理用户输入
- ❌ 绝对不将敏感数据写入日志
- ❌ 绝对不使用 HTTP（必须 HTTPS）

## 必须遵循
- ✅ 所有用户输入必须做 sanitization
- ✅ API 密钥必须通过环境变量引入
- ✅ 所有数据库查询必须使用参数化查询
- ✅ 所有 API 接口必须有鉴权检查
```

#### 使用项目规范文件

```
项目根目录/
├── .agent/
│   └── skills/
│       ├── security-standards/     # 安全规范
│       ├── error-handling/         # 错误处理规范
│       └── performance-rules/     # 性能要求
```

当 AI 在生成代码前就加载了这些规范，很多问题从源头就被避免了。

> 💡 **这就是 Skills 在 Vibe Coding 中最大的价值**：**不是事后救火，而是事前防火。**

## 🎯 「不看代码」的正确理解

回到 Karpathy 的那句话。他说「不看代码」，但他的隐含条件是：

1. 他有 20+ 年的编程经验，能快速判断结果对不对
2. 他做的是个人实验项目，不是金融系统
3. 他能通过运行效果来判断代码质量

所以「不看代码」的正确理解不是「完全不看」，而是——

### 看代码的粒度变了

| 阶段 | 传统方式 | Vibe Coding |
|------|---------|-------------|
| 写代码时 | 逐行编写审视 | 描述意图，看AI生成 |
| 审查代码时 | 逐行 Code Review | 架构层面审查 + 关键路径抽检 |
| 调试时 | Debug 断点逐行跟踪 | 看报错 → 描述问题 → AI 修复 |
| 优化时 | Profiling + 手动优化 | 描述性能问题 → AI 给方案 → 验证效果 |

你从显微镜换成了望远镜。不是你看不见细节了，而是你选择在更高的维度工作。

### 什么时候必须拿起显微镜

- 🔴 涉及用户数据安全时
- 🔴 涉及金钱交易时
- 🔴 涉及核心业务逻辑的正确性时
- 🔴 代码要上生产环境时（至少要跑一遍测试）

## ✅ Vibe Coding 质量清单

每次 Vibe Coding 交付前，过一遍这个 Checklist：

### 🔒 安全
- [ ] 没有硬编码的密钥或凭证
- [ ] 用户输入都做了校验和转义
- [ ] API 接口都有鉴权
- [ ] 敏感数据不在前端暴露

### 🏗️ 架构
- [ ] 代码结构清晰，模块边界明确
- [ ] 没有不必要的复杂度（Kill the Over-Engineering）
- [ ] 关键组件有错误处理和降级策略

### ⚡ 性能
- [ ] 没有明显的 N+1 查询
- [ ] 大列表有分页或虚拟滚动
- [ ] 静态资源有缓存策略
- [ ] 没有内存泄漏的明显迹象

### 🧪 测试
- [ ] 核心功能有自动化测试覆盖
- [ ] 异常路径被测试过
- [ ] 实际运行了测试且全部通过

### 📦 部署
- [ ] 环境变量配置正确
- [ ] 构建产物没有包含开发环境的东西
- [ ] 错误监控已配置（至少有个 console.error 兜底）

---

## 💡 总结

Vibe Coding 的代码质量问题不是无解的。关键在于建立**正确的防线体系**：

```
第三道防线（最重要）：架构约束  ← Skills + 项目规范
    ↓ 从源头减少问题
第一道防线：AI 自检          ← 测试 + 代码审查
    ↓ 自动化兜底
第二道防线：人工抽检          ← 关键路径 Review
    ↓ 把关核心安全
```

**Vibe Coding 不是不负责任地把一切交给 AI，而是用更聪明的方式分配人和 AI 的精力。**

AI 做广度覆盖，人做深度把关。

这就是 2026 年写代码的正确姿势。
