# Vibe Coding 不是「让 AI 写代码」这么简单

> 2025 年末，Andrej Karpathy 发了一条推文，造了一个新词——Vibe Coding。半年过去，这个词已经从社交媒体上的段子变成了一个认真的技术趋势。但大多数人对它的理解还停留在「就是让 AI 帮你写代码嘛」。这篇文章要告诉你：Vibe Coding 远不止如此。

---

## 🎙️ 一条推文引发的范式革命

2025 年 11 月，前 Tesla AI 总监、OpenAI 联合创始人 Andrej Karpathy 在 X（前 Twitter）上发了一条推文：

> *"There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists... I just see stuff, say stuff, run stuff, and copy paste stuff, and it mostly works."*

翻译过来就是：**"有一种新的编程方式叫 Vibe Coding——你完全跟着感觉走，接受指数级增长，忘掉代码本身的存在。我只是看到东西、说出来、跑一下、复制粘贴，然后大部分时候就能用了。"**

这条推文在开发者圈子里炸了。

有人说这是未来，有人说这是偷懒。而真相，比这两种简单判断都要深刻得多。

---

## 🔍 概念拆解：三种编程范式的本质区别

要理解 Vibe Coding，先搞清楚它和另外两个容易混淆的概念有什么不同。

### 传统编程

```
人 → 思考逻辑 → 手写代码 → 调试 → 运行
```

**核心特征**：开发者掌控每一个字符。你脑子里有完整的代码模型，你知道每一行代码在做什么。

### AI 辅助编程（Copilot 模式）

```
人 → 写部分代码 → AI 补全 → 人审核 → 接受/拒绝
```

**核心特征**：AI 是你的副驾驶。你还是在开车，AI 帮你换挡、查导航。你仍然逐行审阅代码，AI 只是让你写得更快。

### Vibe Coding

```
人 → 描述意图 → AI 生成整体方案 → 人判断效果 → 继续迭代或调整方向
```

**核心特征**：你不再逐行读代码。你关注的是**结果对不对**，而不是**代码怎么写的**。

### 关键区别一览

| 维度 | 传统编程 | AI 辅助编程 | Vibe Coding |
|------|---------|-----------|-------------|
| 人的角色 | 编写者 | 审阅者 | 导演/决策者 |
| AI 的角色 | 无 | 打字助手 | 执行团队 |
| 关注点 | 代码实现 | 代码质量 | 产品效果 |
| 控制粒度 | 逐字符 | 逐行 | 逐功能/逐模块 |
| 迭代方式 | 改代码 | 改提示后审阅 | 用自然语言描述意图 |
| 典型心态 | "我来写" | "AI 帮我写" | "我描述，AI 实现" |

## ✨ Vibe Coding 的三个核心特征

### 1. 意图驱动（Intent-Driven）

传统编程你要告诉计算机 **how**（怎么做），Vibe Coding 你只需要告诉 AI **what**（做什么）和 **why**（为什么做）。

```
传统方式：
"创建一个 Express 路由，接收 POST 请求，解析 JSON body，
验证 email 字段格式，对 password 做 bcrypt 加密，
存入 PostgreSQL 的 users 表，返回 JWT token..."

Vibe Coding 方式：
"做一个用户注册功能，要安全，数据存数据库。"
```

不是说 Vibe Coding 你可以描述得很模糊——恰恰相反，**好的意图描述是一种高阶能力**。但你描述的维度变了：从实现细节变成了业务目标和质量标准。

### 2. 对话迭代（Conversational Iteration）

Vibe Coding 的开发过程不是「写代码 → 编译 → 调试」，而是「对话 → 观察 → 反馈 → 继续对话」。

```
你：做一个待办事项页面
AI：（生成了完整页面）

你：不错，但列表太丑了，参考 Linear 的风格改改
AI：（重新设计了列表样式）

你：添加拖拽排序，还有完成时加个动画效果
AI：（加了拖拽和动画）

你：动画太夸张了，改柔和一点，0.3s 的ease-out
AI：（调整了动画参数）
```

整个过程就像在跟一个高效的执行团队开会：你负责方向和品味，他们负责落地。

### 3. 放弃逐行控制（Letting Go of Line-by-Line Control）

这是 Vibe Coding 最颠覆认知的地方。Karpathy 原话说的是「forget that the code even exists」——忘掉代码的存在。

**这不是说代码不重要**，而是说你关注的层次变了：

| 传统关注 | Vibe 关注 |
|---------|----------|
| 这个函数的时间复杂度是什么？ | 页面响应速度够快吗？ |
| 变量命名是否规范？ | 代码整体可维护吗？ |
| 这行逻辑有 edge case 吗？ | 用户流程走得通吗？ |
| API 接口设计是否 RESTful？ | 功能是否满足需求？ |

你从「代码审查员」变成了「产品验收官」。

## 📈 进化路径：从 Copilot 到 Agent 到 Vibe

Vibe Coding 不是凭空冒出来的，它是 AI 编程能力不断提升的自然结果：

```
2021 → GitHub Copilot
        AI 会补全你的代码了！
        ↓
2023 → ChatGPT / Claude / GPT-4
        AI 能写完整函数了！但你还得复制粘贴
        ↓
2024 → Cursor / Claude Code / Gemini
        AI 直接在 IDE 里改你的项目了
        ↓
2025 → Agentic AI
        AI 能自主完成多步骤任务了：
        分析需求 → 写代码 → 跑测试 → 修 Bug → 提 PR
        ↓
2026 → Vibe Coding 成为主流
        开发者开始习惯用意图驱动的方式做项目
        人的角色从"编写者"转变为"决策者"
```

每一步都在扩大 AI 的自治边界，缩小人的直接操控范围。Vibe Coding 是这条曲线的最新一个拐点。

## 👥 谁在 Vibe Coding？

一个常见的误解是：Vibe Coding 只适合编程新手或不想写代码的人。

事实恰恰相反。

### 资深开发者的新姿势

真正在大量使用 Vibe Coding 的反而是资深开发者。原因很简单：

1. **他们知道什么是好的代码**——所以他们能判断 AI 生成的结果是否合格
2. **他们知道什么时候该干预**——当 AI 走偏时，他们能精准介入
3. **他们的时间更贵**——用 Vibe Coding 做 80% 的工作，把精力花在最关键的 20% 上

### 一个真实的场景

> 一个有 10 年经验的全栈工程师，用 Vibe Coding 在一个周末做了一个 AI 推理游戏。如果用传统方式，这至少需要两到三周。他并不是不会写代码——他太会写了，所以他知道哪些代码不值得自己手写。

### 初学者也能 Vibe

Vibe Coding 确实降低了入门门槛。一个设计师、产品经理、甚至学生，也能用 Vibe Coding 做出可用的应用。这不是坏事——更多人能把想法变成现实，只会让整个生态更繁荣。

**但要注意**：初学者做 Vibe Coding 的风险比资深开发者高。因为你可能不知道生成的代码里藏着什么坑。这也是为什么后续文章会专门讲「质量红线」。

## 🧭 Vibe Coding 不是偷懒

最后回到一个核心问题：Vibe Coding 是不是就是在偷懒？

**不是。**

Vibe Coding 是一种**编程范式转变**。就像：

- 从汇编语言到高级语言不是偷懒——是抽象层次提升了
- 从手写 CSS 到用 Tailwind 不是偷懒——是工作方式进化了
- 从手写 SQL 到用 ORM 不是偷懒——是关注点转移了

Vibe Coding 本质上做的是同一件事：**把人的注意力从低层次的实现拉到高层次的决策上。**

你不再花时间想「这个循环该用 for 还是 forEach」，而是花时间想「这个功能到底应该怎么设计才对用户最友好」。

这不是偷懒，这是**更高效地使用人类最宝贵的资源——创造力和判断力**。
